## SECTION 2 — ARCHITECTURE PHILOSOPHY

### §2.1 The Magnum Opus Statement

Poseidon Mode is the true form of the Trident Agent. Every tool, every hook,
every semantic-intelligence subsystem, every audit layer (R0–R16), every
context-synthesis stage, and every principle (P1–P10) exists for one reason:
so that Poseidon Mode can function as **GOD MODE**.

GOD MODE means a single autonomous execution path that:

```
1. Takes a target project path as input.
2. Audits the codebase with full AST-grade intelligence.
3. Scores the codebase against a deterministic runtime-grade standard.
4. Decides what to fix and in what order.
5. Plans remediation waves with dependency-aware DAG resolution.
6. Dispatches parallel build-agent waves to execute the fixes.
7. Collects results, re-audits, and re-scores.
8. Loops until score >= 96 (the LOCKED setpoint) or a hard terminal is hit.
9. Runs for 9+ hours unattended if necessary.
10. Resumes with full context after every compaction.
```

Poseidon is not a feature. It is the **telos** of the entire Trident platform. If
Poseidon Mode cannot run autonomously for 9+ hours and deliver a runtime-grade
codebase, then every other subsystem has failed its purpose. v4.4.1 exists to
restore the architecture that makes this possible, after v4.4.1-rc and v4.4.2
demonstrated exactly how to break it.

### §2.2 The Central Thesis: Algorithmic, Not Model-Dependent

From `00_PHILOSOPHY.md`, the founding axiom of the Trident platform:

> "agent harnesses should be algorithmic, not model-dependent."

This is not a slogan. It is an engineering constraint with a precise meaning:

```
DEFINITION — Algorithmic, Not Model-Dependent:
  The Poseidon loop MUST converge to score >= 96 regardless of which
  language model drives it. A frontier model and a mid-tier model must
  both complete the loop. The convergence is guaranteed by MECHANICAL
  infrastructure (state files, hash scoring, tool-returned instructions),
  not by the model's cleverness.
```

Why this matters: models drift. Models get quantized. Models get swapped for
cheaper ones. Models hallucinate, lose context, and derail. If the loop's
correctness depends on the model being smart, then the loop is fragile. The
correct architecture makes the loop correct *in spite of* the model, and uses the
model only as a generic actuator that reads instructions and executes tool calls.

The infra must work with ANY model — dumb or genius. A genius model will reach
score 96 in fewer cycles. A dumb model will reach score 96 in more cycles. But
BOTH reach score 96, because the state machine, the scoring, and the instruction
flow are mechanical and model-independent.

### §2.3 The Three Pillars of Anti-Derailment

Autonomous 9-hour execution is impossible without systematic anti-derailment.
Trident v4.4.1 rests on three pillars. ALL THREE are required. Remove any one
and the loop collapses.

#### Pillar 1 — Mechanical Algorithmic Infrastructure

```
state.json          Atomic phase/transition persistence (P5: snapshot-and-rollback)
hook enforcement    tool.execute.after fires regardless of model cooperation
evidence gate       passRate >= 0.96 enforced by hash comparison, not opinion
hash scoring        findings reduced to SHA-256 signatures, compared deterministically
SQLite WAL          durable session/gate/guardian/identity/audit tables
Merkle chain        every evidence artifact linked by SHA-256, verifyChain() pre-transition
```

These subsystems do not ask the model to cooperate. They observe the model's
actions through tool-call telemetry and enforce correct behavior mechanically.
The model cannot lie to a hash. The model cannot talk its way past a gate that
compares two SHA-256 digests.

#### Pillar 2 — Real Intelligence in the Tools

The intelligence of Trident is not in the model. It is in the tools. Each tool
performs real computation, not narrative:

```
trident-code-audit        18-layer AST analysis (R0–R16), regex + TS Compiler API
trident-deep-planning     3-layer: first-principles → workflow → context-library
trident-context-synthesis 4-layer: collect → score → compress → inject (T1/T2)
trident-problem-solving   6-layer: assumption→action→observe→gap→meta→verify
```

These tools NEVER guess. `trident-code-audit` walks the TypeScript AST and emits
findings with file, line, severity, confidence, and a concrete correction.
`trident-deep-planning` resolves a dependency DAG and emits a wave manifest.
`trident-context-synthesis` scores context fragments for relevance and compresses
them into injectable T1 or standalone T2 artifacts. `trident-problem-solving`
applies a 6-layer mental framework to diagnose WHY a score stalled.

The model's job is to CALL these tools and ACT ON their explicit text output.
The tools are the driver. The model is the engine.

#### Pillar 3 — Runtime Grade Standards

```
P1–P10 principles         Every line of new code must satisfy these (see §6.1)
Theatrical detection      Order 5 classifier catches fake/empty/stub handlers
Evidence chain             SHA-256 Merkle chain, verifyChain() before every transition
Iron Laws IL-01..IL-12    Common-sense T3 invariants (see §6.3)
7 Questions                Answered before writing any function (see §6.5)
```

These standards are what make the output "runtime grade" rather than "looks
plausible." A build wave that produces theatrical code (empty handlers, no-op
catches, mock suggestions) is quarantined by the VERIFY phase and never advances
the score. The standards are enforced by the audit tools, not by the model's
conscience.

### §2.4 Tool-First Execution

The single most important architectural decision in Trident is **tool-first
execution**. Its precise meaning:

```
DEFINITION — Tool-First Execution:
  Every decision in the Poseidon loop is backed by a tool call.
  The tool returns EXPLICIT TEXT INSTRUCTIONS for the current phase.
  The model reads those instructions and executes them mechanically.
  The model never narrates a plan without first calling the tool that
  produces the plan. The model never claims a fix without first calling
  the tool that verifies the fix.
```

Tool-first execution makes ANY model 100x smarter, because the model is no longer
reasoning from its parametric memory — it is reasoning from structured tool
output that was computed against the actual codebase. A mid-tier model reading
"the AUDIT phase found 14 findings, here are their signatures, now enter SCORE"
will execute correctly. The same model asked to "figure out the audit yourself"
will hallucinate.

This is the lesson that v4.4 internalized and that v4.4.1-rc/v4.4.2 forgot.

### §2.5 The Lesson from v4.4

The v4.4 session (`ses_11548d`) proved the architecture. In that session, the
agent demonstrated behavior that an observer would call AGI-level: it autonomously
audited a codebase, identified the real defects, planned remediation, dispatched
build waves, and converged.

The critical observation: **no hook drove the loop.** The loop was driven by the
`trident-poseidon` tool, which returned explicit, phase-specific text instructions
on every invocation. The model read those instructions and acted on them. The
tool was the driver; the model was the engine; the state file was the memory.

The v4.4 architecture was:

```
Model  →  calls trident-poseidon
Tool   →  reads state.json, advances phase, returns explicit instructions
Model  →  reads instructions, calls the specified tool (code-audit, deep-planning, etc.)
Tool   →  computes real result, writes evidence
Model  →  calls trident-poseidon again → next phase
```

This worked. It worked so well that the session ran for hours and converged. The
lesson: **the tool-based instruction flow is the correct architecture. Do not
replace it.**

### §2.6 The Lesson from v4.4.1-rc and v4.4.2 — DO NOT REPEAT

Both v4.4.1-rc and v4.4.2 were disasters. They failed for the SAME root cause:
they REPLACED the working tool-based architecture with a **hook state machine**.

The specific failure mode:

```
v4.4.1-rc / v4.4.2 anti-pattern:
  1. Built a god-loop-hook.ts (~430 lines) that ran the state machine IN THE HOOK.
  2. The hook injected DYNAMIC content into the system prompt every turn.
     → This destroyed the prompt cache (KV cache invalidated every turn).
     → Performance collapsed; token costs exploded.
  3. The hook REPLACED accurate extractScore() with a filesystem-diff heuristic.
     → Score became an opinion, not a measurement. Convergence broke.
  4. The hook REPLACED the tool-based instruction flow with mechanical gates.
     → The model no longer received explicit instructions. It received
       opaque "you are derailed" signals with no actionable content.
  5. Added 4 new helper files (activation-parser.ts, wave-manifest.ts, etc.)
     that added no functional value and fragmented the logic.
```

The result: a loop that could not converge, could not resume after compaction,
could not maintain a cache, and could not score accurately. v4.4.2 was abandoned.

v4.4.1 is the CORRECTION. It restores the v4.4 tool-based architecture, keeps the
orchestrator class as the loop driver, reduces the hook to a thin model-independent
resilience layer (~120 lines, static injection only), and forbids every anti-pattern
listed in §7.

```
THE INVARIANT:
  THE MODEL IS THE ENGINE.
  THE TOOL IS THE DRIVER.
  THE STATE FILE IS THE MEMORY.
  THE HOOK IS A GUARDRAIL, NOT A DRIVER.
```

### §3.1 Layer 1 — Intelligence (GodLoopOrchestrator + trident-poseidon)

The `GodLoopOrchestrator` class STAYS. It is enhanced, NOT replaced. The
10-phase state machine runs INSIDE the orchestrator class. The orchestrator is
instantiated once per Poseidon session and lives for the entire 9+ hour run.

The `trident-poseidon` tool is the **driver**. Every time the model calls
`trident-poseidon`, the tool:

```
1. Loads state.json from disk (or initializes if absent).
2. Validates the evidence chain (verifyChain()).
3. Determines the current phase.
4. Executes the mechanical action for that phase
   (call code-audit, compute score, call deep-planning, etc.).
5. Writes the new state atomically to state.json.
6. Returns EXPLICIT TEXT INSTRUCTIONS to the model for what to do next.
```

The model NEVER has to guess what phase it is in or what to do. The tool tells
it, in plain text, every single turn.

| Phase          | Tool/System Called              | Mechanical Action                                        | State Written                          | Text Instruction Returned to Model                                                                |
|----------------|---------------------------------|----------------------------------------------------------|----------------------------------------|---------------------------------------------------------------------------------------------------|

The model acts on it. This is tool-first execution.

### §3.2 Layer 2 — Enforcement (poseidon-enforcer-hook.ts)

The enforcement hook is a **thin, model-independent resilience layer**. It is
~120 lines, NOT 430. It does NOT drive the loop. It guards against derailment.

#### §3.2.1 Hook Contract

```typescript
// FILE: src/hooks/poseidon-enforcer-hook.ts
// LINES: ~120 (hard ceiling; if it grows beyond 150, it is WRONG)
// FIRES ON: tool.execute.after
// NEVER FIRES ON: tool.execute.before (the orchestrator owns pre-execution)

export interface PoseidonEnforcerConfig {
  stateJsonPath: string;        // absolute path to .trident/state.json
  enabled: boolean;             // true only when mode === 'poseidon'
  maxDerailmentsBeforeLockout: number;  // default 4
  derailmentDecayInterval: number;      // every N successful actions, decrement by 1
}

export interface EnforcerAction {
  type: 'INFORM' | 'WARN' | 'BLOCK' | 'RESTART' | 'LOCKOUT';
  message: string;              // STATIC template, no dynamic prompt injection
  injectInto: 'messages';       // ALWAYS 'messages', NEVER 'system'
  counterAfter: number;         // derailment count after this action
}
```

The hook reads `state.json` from disk on every `tool.execute.after` event. It
checks whether the tool that just executed matches the **expected tool for the
current phase** (the phase-to-expected-tool mapping from §3.1.1). If the model
called the wrong tool, or called no tool and instead narrated, the hook escalates.

#### §3.2.2 STATIC vs DYNAMIC Injection — The Cache Rule

```
RULE (CACHE SAFETY — BINDING):
  The system prompt is injected ONCE, statically, at session start:
    "Poseidon Mode active. Call trident-poseidon to advance."
  This is a SINGLE LINE. It never changes. The KV cache is preserved across turns.

  DYNAMIC phase detail (current phase, score, instructions) is returned by the
  trident-poseidon TOOL in the MESSAGES channel — NOT in the system prompt.
  Tool output is appended to the conversation, never to the system prompt.

  VIOLATION OF THIS RULE = prompt cache destruction = v4.4.2 regression.
  This is Banned Pattern #2 (see §7).
```

#### §3.2.3 Escalation Ladder

The escalation ladder is deterministic. Counters NEVER reset within a session.
Counters decay: every 5 consecutive successful (on-track) actions decrements the
derailment counter by 1, floor 0.

```
First derailment   → WARN
  Action: log to guardian_log; inject STATIC correction into messages:
          "Off-track. Current phase is <PHASE>. Call <EXPECTED_TOOL>. (warn #1)"
  The hook does NOT reset state. The orchestrator owns state.

Second derailment  → BLOCK
  Action: log; inject stronger STATIC correction:
          "Repeated off-track behavior. You MUST call <EXPECTED_TOOL> now. (block #2)"
  Still no state mutation by the hook.

Third derailment   → RESTART
  Action: log; the hook requests the orchestrator to reset to the last
          CHECKPOINT phase. The orchestrator (not the hook) performs the reset.
          The hook emits: "Phase reset to <CHECKPOINT_PHASE>. Resume from there. (restart #3)"

Fourth+ derailment → LOCKOUT
  Action: log; emit:
          "LOCKOUT. Derailment threshold exceeded. Pausing. External intervention required. (lockout)"
  The loop pauses. No further tool calls are accepted until a human operator
  issues an explicit resume command or bumps the lockout ceiling.
```

```typescript
// Escalation decision (pure function, no side effects beyond logging)
function escalate(derailmentCount: number): EnforcerAction['type'] {
  if (derailmentCount >= 4) return 'LOCKOUT';
  if (derailmentCount === 3) return 'RESTART';
  if (derailmentCount === 2) return 'BLOCK';
  return 'WARN'; // count === 1
}

// Decay: called after every successful on-track action
function decayDerailments(current: number, successesSinceLastDecay: number): number {
  if (successesSinceLastDecay >= 5 && current > 0) {
    return current - 1;  // decrement by 1, floor 0
  }
  return current;
}
```

The hook is model-independent: it works with ANY model because it keys off the
**tool name** that was executed (objective telemetry), not off narrative content
(subjective parsing). A genius model and a dumb model that both call the wrong
tool both get the same escalation.

### §3.3 Layer 3 — Memory (state.json + context-synthesis + SQLite)

Memory is what lets Poseidon run for 9+ hours and survive compaction. There are
three memory subsystems, each with a distinct role.

#### §3.3.1 state.json — Atomic Phase Memory

`state.json` lives at `<targetPath>/.trident/state.json`. It is the orchestrator's
working memory. Every phase transition writes it **atomically** (P5: snapshot-and-
rollback — write to temp file, fsync, rename).

```typescript
// Atomic write pattern (P5 compliant)
function writeStateAtomic(path: string, state: GodLoopState): void {
  const tmp = path + '.tmp';
  const data = JSON.stringify(state, null, 2);
  writeFileSync(tmp, data, { encoding: 'utf-8' });
  fsyncSync(openSync(tmp, 'r'));
  renameSync(tmp, path);  // atomic on same filesystem
}
```

If a write fails mid-operation, the previous state.json is intact (the rename
never happened). The orchestrator can always recover from the last known good
state.

#### §3.3.2 context-synthesis — T1/T2 Context Bridge

After each phase (and especially after each wave), `trident-context-synthesis`
generates context artifacts that survive compaction:

```
T1 injectable  (lightweight, ~2K tokens)
  Generated after EVERY phase/wave.
  Contains: current phase, score, wave number, last action, next instruction.
  Injected into the conversation as a compact context block.
  Mix target: 60–70% of context budget.

T2 knowledge file (dense, standalone, ~10–20K tokens)
  Generated at MILESTONES (score milestone, wave completion, phase checkpoints).
  Written to disk: <targetPath>/.trident/context/T2_<milestone>.md
  Contains: full reasoning chain, audit findings, decisions, evidence hashes.
  Mix target: 40–30% of context budget.
```

On compaction (the runtime collapses the conversation to save tokens), the
orchestrator re-injects: latest T1 + most recent T2 + a literal "continue"
instruction. The agent resumes with full context. This is how Poseidon spans
15M+ output tokens across a single logical session.

```sql
  token_estimate INTEGER NOT NULL DEFAULT 0,
  hash        TEXT NOT NULL,                 -- SHA-256 of artifact content
  created_at  INTEGER NOT NULL DEFAULT (unixepoch())
);

-- TABLE 5: audit_trail — IMMUTABLE append-only log (Merkle-linked)
CREATE TABLE IF NOT EXISTS audit_trail (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id    TEXT NOT NULL,
  timestamp     INTEGER NOT NULL DEFAULT (unixepoch()),
  event         TEXT NOT NULL,               -- phase_transition|evidence|gate|guardian
  phase         TEXT,
  prev_hash     TEXT NOT NULL,               -- SHA-256 of previous row (Merkle link)
  this_hash     TEXT NOT NULL,               -- SHA-256 of (prev_hash || payload)

CREATE INDEX IF NOT EXISTS idx_audit_trail_session ON audit_trail(session_id, timestamp);
CREATE INDEX IF NOT EXISTS idx_guardian_session ON guardian_log(session_id, timestamp);
```

The `audit_trail` table is the Merkle chain substrate. Each row's `this_hash` is
`SHA-256(prev_hash || payload)`. The genesis row has `prev_hash = SHA-256('genesis')`.
`verifyChain()` walks the table from genesis and recomputes every hash. Any
tampering breaks the chain and fails verification.

```typescript
// Merkle chain verification (called before EVERY state transition)
function verifyChain(db: Database, sessionId: string): boolean {
  const rows = db.prepare(
    'SELECT prev_hash, this_hash, payload FROM audit_trail WHERE session_id = ? ORDER BY id ASC'
  ).all(sessionId) as AuditRow[];

  let prevHash = sha256('genesis');
  for (const row of rows) {
    if (row.prev_hash !== prevHash) return false;          // chain break
    const computed = sha256(prevHash + row.payload);
    if (computed !== row.this_hash) return false;           // tamper detected
    prevHash = row.this_hash;
  }
  return true;
}
```

```
                                  LOCKED ◄──────────── DISPATCH ◄───────┘
                                  (terminal)             │
                                                         ▼
                                            COLLECT ──► VERIFY
                                                         │
                                                         ▼
                                                  AUDIT_RECHECK ──► SCORE (cycle++)
                                                         │
                                                         │ (loop back to DECIDE)

      DECIDE ──► stall (no improvement 3 cycles) ──► PROBLEM_SOLVE ──► PLAN
      DECIDE ──► cycle > maxCycles (50)           ──► FAILED (terminal)
```

```
Terminal states:
  LOCKED   — score >= 96 reached. Success. Loop halts.
  FAILED   — cycle > 50 reached without convergence. Loop halts.
```

#### §4.4.2 AUDIT

`trident-code-audit` is invoked with `action='full'`. Bundle exclusion is applied
(node_modules, dist, .git, *.lock). The 18-layer engine (R0–R16) walks every .ts
file via the TypeScript Compiler API. Each finding carries a confidence score;
findings below the **0.85 confidence floor** are discarded (they are noise, not
signal). Each surviving finding is reduced to a `FindingSignature` (SHA-256 of
file+line+layer+rule) and stored in `preAuditFindings[]`.

```typescript
function phaseAudit(state: GodLoopState): GodLoopState {
  const result = tridentCodeAudit({
    action: 'full',
    target: state.targetPath,
    exclude: ['**/node_modules/**', '**/dist/**', '**/*.lock'],
    confidenceFloor: 0.85,
  });
  const findings: FindingSignature[] = result.findings
    .filter((f) => f.confidence >= 0.85)
    .map((f) => ({
      file: f.file,
      line: f.line,
      layer: f.layer,
      severity: f.severity,
      confidence: f.confidence,
      signature: sha256(`${f.file}:${f.line}:${f.layer}:${f.rule}`),
      correction: f.correction,
    }));
  const evidenceRootHash = sha256(findings.map((f) => f.signature).sort().join(''));
  return { ...state, phase: 'AUDIT', preAuditFindings: findings, evidenceRootHash };
}
```

**Text instruction returned:**
```
"AUDIT complete. <N> findings (confidence >= 0.85). Critical: <C>, High: <H>, Medium: <M>, Low: <L>.
Evidence root hash: <evidenceRootHash>. Enter SCORE: call trident-poseidon to compute score."
```

#### §4.4.3 SCORE

The proven `extractScore()` function (regex-based, deterministic) computes the
score by comparing `preAuditFindings` to `postAuditFindings`. On the first pass
(cycle 0), `postAuditFindings` is empty, so the score reflects the raw defect
density. On subsequent passes, the score reflects how many pre-wave findings were
resolved.

```typescript
const TYPE_WEIGHT: Record<string, number> = {
  '.ts': 1.0,
  '.js': 0.1,
  '.tsx': 1.0,
  '.jsx': 0.1,
};

function extractScore(
  pre: FindingSignature[],
  post: FindingSignature[],
  cycle: number,
): number {
  if (pre.length === 0) return 100;          // nothing to fix
  const preSet = new Set(pre.map((f) => f.signature));
  const postSet = new Set(post.map((f) => f.signature));
  const resolved = [...preSet].filter((s) => !postSet.has(s)).length;
  const remaining = preSet.size - resolved;
  // Progressive: weight resolution; penalize remaining criticals heavily
  const resolutionRatio = resolved / preSet.size;
  const criticalRemaining = post.filter(
    (f) => f.severity === 'critical' || f.severity === 'high',
  ).length;
  let score = Math.round(resolutionRatio * 100);
  score -= criticalRemaining * 5;             // each remaining critical/high: -5
  score = Math.max(0, Math.min(100, score));
  return score;
}
```

**Text instruction returned:**
```
"SCORE computed: <score>/100 (cycle <cycle>). Resolved <resolved>/<preTotal> findings.
Remaining critical/high: <criticalRemaining>. Enter DECIDE."
```

#### §4.4.4 DECIDE

Pure logic. No tool call. The decision tree:

```typescript
function phaseDecide(state: GodLoopState, maxCycles = 50): GodLoopState {
  const { score, highestScore, cycle, stalledSince } = state;
  // 1. Convergence
  if (score >= 96) {
    return { ...state, phase: 'LOCKED', highestScore: Math.max(highestScore, score) };
  }
  // 2. Hard terminal
  if (cycle > maxCycles) {
    return { ...state, phase: 'FAILED', error: `Exceeded maxCycles (${maxCycles})` };
  }
  // 3. Stall detection: no improvement for 3 consecutive cycles
  const stalled = score <= highestScore;
  const newStalledSince = stalled ? stalledSince + 1 : 0;
  if (newStalledSince >= 3) {
    return { ...state, phase: 'PROBLEM_SOLVE', stalledSince: newStalledSince };
  }
  // 4. Normal: plan next wave
  return {
    ...state,
    phase: 'PLAN',
    highestScore: Math.max(highestScore, score),
    stalledSince: newStalledSince,
  };
}
```

**Text instruction returned (examples):**
```
score >= 96: "DECIDE: score >= 96. Convergence reached. Enter LOCKED."
stall:       "DECIDE: stalled for 3 cycles (score flat at <score>). Enter PROBLEM_SOLVE."
cycle > 50:  "DECIDE: cycle <cycle> exceeds maxCycles (50). Enter FAILED."
normal:      "DECIDE: score < 96, not stalled, cycle <cycle>/50. Enter PLAN."
```

#### §4.4.5 PLAN

`trident-deep-planning` is invoked. It resolves a dependency DAG over the files
referenced in the unresolved findings, groups them into waves (topological
order — a file's dependencies must be fixed before the file itself), and emits a
`WaveManifest` with per-agent specs.

```typescript
function phasePlan(state: GodLoopState): GodLoopState {
  const unresolved = state.preAuditFindings.filter(
    (f) => !state.postAuditFindings.some((p) => p.signature === f.signature),
  );
  const plan = tridentDeepPlanning({
    targetPath: state.targetPath,
    findings: unresolved,
    failures: [],                    // empty unless coming from PROBLEM_SOLVE
    contextT1: state.contextBridgePath,
  });
  const manifest: WaveManifest = {
    wave: state.wave + 1,
    agentCount: plan.agents.length,
    agents: plan.agents.map((a) => ({
      agentId: a.id,
      agentType: 'trident_build',
      targetFiles: a.files,
      findings: unresolved.filter((f) => a.files.includes(f.file)),
      contextT1Path: state.contextBridgePath ?? '',
      instructions: a.instructions,
    })),
    dependencyDag: plan.dag,
    preWaveHash: state.snapshotHash,
    estimatedComplexity: plan.complexity,
  };
  return { ...state, phase: 'PLAN', wave: manifest.wave, waveManifest: manifest };
}
```

#### §4.4.6 DISPATCH

The model calls `task()` N times in parallel (one per agent in the wave
manifest). Each agent receives its `WaveAgentSpec` (target files, findings, T1
context path, explicit instructions). The orchestrator does NOT execute the
agents itself — it instructs the model to do so. This is tool-first: the tool
returns the dispatch instruction; the model performs the dispatch.

```typescript

  if (!state.waveManifest) {
    throw new Error('P2 VIOLATION: waveManifest is null at DISPATCH');
  }
  return { ...state, phase: 'DISPATCH' };
}
```

**Text instruction returned:**
```

Each agent receives its WaveAgentSpec. Do not serialize. After all agents return, enter COLLECT."
```

#### §4.4.7 COLLECT

Results from all wave agents are aggregated. `trident-context-synthesis` is
called with `outputMode='T1'` to produce a lightweight injectable for the next
wave. If a milestone was reached (e.g., a full layer cleared, or score crossed a
decile boundary), a T2 knowledge file is also generated.

```typescript
function phaseCollect(state: GodLoopState, agentResults: AgentResult[]): GodLoopState {

}
```

#### §4.4.8 VERIFY

`evidenceStore.verifyChain()` is called (Merkle chain walk, §3.3.4). The pass
rate must be >= 0.96. A content hash diff is computed on all files the wave
claimed to modify — if a file's hash is unchanged but the agent claimed a fix,
that is a **theatrical detection** (Order 5) and the artifact is quarantined.

```typescript
function phaseVerify(state: GodLoopState): { state: GodLoopState; quarantined: boolean } {
  const chainValid = verifyChain(getDb(), state.targetPath);
  if (!chainValid) {
    return {
      state: { ...state, phase: 'VERIFY', error: 'Merkle chain verification failed' },
      quarantined: true,
    };
  }
  // Content hash diff: did the files actually change?
  const modifiedFiles = state.waveManifest?.agents.flatMap((a) => a.targetFiles) ?? [];
  const theatrical = modifiedFiles.filter((f) => {
    const before = hashFileAtSnapshot(state.snapshotHash, f);
    const after = hashFile(`${state.targetPath}/${f}`);
    return before === after; // unchanged = theatrical claim
  });
  if (theatrical.length > 0) {
    // Quarantine: do not advance score; re-plan the wave
    logQuarantine(theatrical);
    return { state: { ...state, phase: 'PLAN' }, quarantined: true };
  }
  const postWaveHash = computeDirectoryHash(state.targetPath, modifiedFiles);
  return { state: { ...state, phase: 'VERIFY', postWaveHash }, quarantined: false };
}
```

**Text instruction returned:**
```
pass:    "VERIFY passed. Chain valid. No theatrical code. Enter AUDIT_RECHECK."
fail:    "VERIFY FAILED. <reason>. Theatrical files: [...]. Re-entering PLAN to retry wave."
```

#### §4.4.9 AUDIT_RECHECK

`trident-code-audit` is re-run on the **modified files only** (not the full
codebase — this is faster and focuses on the wave's effect). The resulting
findings become `postAuditFindings[]`. The phase transitions to SCORE, which
compares pre/post and produces the new score.

```typescript
function phaseAuditRecheck(state: GodLoopState): GodLoopState {

    target: state.targetPath,

    exclude: ['**/node_modules/**', '**/dist/**'],

    .filter((f) => f.confidence >= 0.85)
    .map((f) => ({
      file: f.file,
      line: f.line,
      layer: f.layer,
      severity: f.severity,
      confidence: f.confidence,
      signature: sha256(`${f.file}:${f.line}:${f.layer}:${f.rule}`),

}
```

**Text instruction returned:**
```
"AUDIT_RECHECK complete. Post-wave findings: <N>. Enter SCORE: recompute score with pre/post comparison."
```

#### §4.4.10 PROBLEM_SOLVE

Triggered when the score has stalled for 3 consecutive cycles. `trident-problem-
solving` applies its 6-layer mental framework (assumption → action → observation
→ gap analysis → meta-reflection → verification) to diagnose WHY the score
stopped improving. Common diagnoses: findings are interdependent and can't be
fixed independently; the build agents are introducing new defects while fixing
old ones; a class of finding is being mis-scored.

```typescript
function phaseProblemSolve(state: GodLoopState): GodLoopState {
```

## SECTION 5 — CLOSED-LOOP CONTROL SYSTEM DIAGRAM

Poseidon Mode is a **closed-loop control system**. Understanding it as such
makes the convergence guarantee intuitive.

### §5.1 Control System Components

```
SETPOINT       score >= 96 (the LOCKED target)
ERROR SIGNAL   error = 96 - score (drives the next wave's intensity)
SENSOR         trident-code-audit (measures the codebase's defect state)
CONTROLLER     GodLoopOrchestrator + DECIDE phase (computes the correction)
ACTUATOR       build waves (DISPATCH → trident_build agents execute fixes)
FEEDBACK       AUDIT_RECHECK → SCORE (re-measures after the actuator acts)
DISTURBANCE    new defects introduced by build agents (rejected by VERIFY)
```

### §5.2 Feedback Loop Diagram

```
                    ┌──────────────────────────────────────────┐
                    │              SETPOINT = 96               │
                    └──────────────────┬───────────────────────┘
                                       │
                                       ▼  (−)
                    error = 96 − score ◄─────────────┐
                                       │              │
                                       ▼              │
                    ┌──────────────────────────────┐  │
                    │       CONTROLLER             │  │
                    │  GodLoopOrchestrator         │  │
                    │  (DECIDE → PLAN)             │  │
                    └──────────────┬───────────────┘  │
                                   │                  │
                                   ▼                  │
                    ┌──────────────────────────────┐  │
                    │       ACTUATOR               │  │
                    │  DISPATCH (build wave)       │  │
                    │  trident_build agents        │  │
                    └──────────────┬───────────────┘  │
                                   │                  │
                                   ▼                  │
                    ┌──────────────────────────────┐  │
                    │   COLLECT → VERIFY            │  │  (disturbance
                    │   (quarantine theatrical)     │  │   rejected here)
                    └──────────────┬───────────────┘  │
                                   │                  │
                                   ▼                  │
                    ┌──────────────────────────────┐  │
                    │       SENSOR (feedback)       │  │
                    │  AUDIT_RECHECK → SCORE        │  │
                    │  trident-code-audit           │  │
                    └──────────────┬───────────────┘  │
                                   │                  │
                                   └──────────────────┘
                                     measured score
                                      (feedback)
```

### §5.3 Control Properties

```
STABILITY     The loop is stable because the actuator only acts on verified
              measurements (VERIFY rejects theatrical output before it can
              corrupt the feedback signal). A disturbance (new defect) is
              detected at the next AUDIT_RECHECK and becomes a new error
              signal — the controller corrects it in the next wave.

CONVERGENCE   Convergence is guaranteed because score is monotonic within a
              non-stalled regime: each wave can only RESOLVE findings (raising
              score) or LEAVE them (score unchanged). Theatrical claims are
              quarantined and never raise the score. The only failure mode is
              a stall, which triggers PROBLEM_SOLVE to break the stall.

ROBUSTNESS    The loop is robust to model quality because the controller logic
              is mechanical (decision tree, hash comparison). A weaker model
              takes more cycles but cannot prevent convergence, because the
              enforcer hook keeps it on-track and the orchestrator's state
              machine is invariant.
```

## SECTION 6 — COMPLIANCE REQUIREMENTS

Every line of new code produced by a build wave MUST satisfy the following
checklist. Build agents verify each item before claiming a task complete. The
VERIFY phase re-checks via the audit tools.

### §6.1 P1–P10 Principles

```
P1  DEFENSIVE IMPORT        Verify a module exists before use. Never assume
                             an import resolved.
P2  TYPE CERTAINTY          Validate at boundaries. NEVER use `as` cast
                             without a runtime guard.
P3  ERROR COMPLETENESS      catch{} with no log/recover/propagate = DEFECT.
                             Every catch MUST log+recover or log+propagate.
P4  RESOURCE LIFECYCLE      setInterval→clearInterval in finally. open→close.
                             Every acquired resource is released.
P5  ATOMIC STATE            Snapshot-and-rollback. Single assignment per
                             transition. Write to temp, fsync, rename.
P6  DEPENDENCY CHECK        if(typeof fs?.readFileSync !== 'function') throw.
                             Verify the API shape before calling it.
P7  PATH RESOLUTION         path.join(os.homedir(), ...). NEVER hardcode
                             /home/ or any absolute user path.
P8  CONFIG VALIDATION       Type check + range check before use. A config
                             value is never trusted until validated.
P9  ASYNC DISCIPLINE        await in try/catch OR .catch(). NEVER
                             fire-and-forget. Every promise is handled.
P10 OUTPUT CONTRACT         Return type accurate in ALL paths, including
                             error paths. No implicit undefined.
```

### §6.2 Iron Laws 1–10 (Semantic Software Engineering Bible)

```
IL-1   The code is the spec. If the code and the doc disagree, the code wins
        and the doc is a lie until corrected.
IL-2   No theatrical code. Empty handlers, no-op catches, stub returns are
        defects, not placeholders.
IL-3   Every function has a contract. Inputs, outputs, errors — documented
        and enforced.
IL-4   Complexity is debt. The simpler implementation that works is correct.
IL-5   Tests describe behavior, not implementation. Test the contract.
IL-6   Dependencies are liabilities. Minimize them. Pin them. Audit them.
IL-7   Logs are for humans. Make them actionable. No log spam.
IL-8   Fail fast, fail loud. Silent failure is the worst failure.
IL-9   State is the enemy. Prefer pure functions. Isolate mutation.
IL-10  The build is the truth. If it doesn't compile and pass tests, it
        doesn't exist.
```

### §6.3 IL-01 through IL-12 (Common Sense T3 Knowledge)

```
IL-01  Verify before you trust. Every external input is hostile until checked.
IL-02  Boundaries are where bugs live. Validate at every system edge.
IL-03  Never trust a cast. Runtime-check the shape, then narrow the type.
IL-04  Errors are data. Handle them explicitly, never swallow.
IL-05  Resources are finite. Close what you open. Clear what you set.
IL-06  State transitions are atomic or they are bugs.
IL-07  Concurrency requires discipline. One writer, many readers, or locks.
IL-08  Naming is documentation. A good name replaces a comment.
IL-09  The happy path is the last thing you write, not the first.
IL-10  Cleanup runs in finally. Always. No exceptions.
IL-11  Hashes don't lie. When in doubt, compare digests.
IL-12  The simplest correct thing is the best thing.
```

### §6.4 Evidence Requirements (R1–R3)

```
R1  DETERMINISTIC OUTPUT    Every tool output is reproducible. Same input →
                            same output. No randomness, no time-of-day
                            variation, no model-opinion leakage.
R2  EVIDENCE PRODUCTION     Every claim produces a filesystem artifact or a
                            hash. No claim is accepted on narration alone.
                            "I fixed it" without a hash diff = rejected.
R3  SAFE FAILURE            Every failure produces a logged, recoverable
                            error with context. No silent corruption. The
                            loop halts cleanly (LOCKED/FAILED) rather than
                            spinning on broken state.
```

### §6.5 The 7 Questions (Answer Before Writing Any Function)

Before a build agent writes ANY function, it must answer all seven:

```
Q1  What are the exact inputs (types, ranges, nullability)?
Q2  What are the exact outputs (type, shape, all paths including errors)?
Q3  What can fail, and how is each failure handled (P3)?
Q4  What resources are acquired, and where are they released (P4)?
Q5  What state is mutated, and is the mutation atomic (P5)?
Q6  What are the boundary conditions, and are they validated (P2/P8)?
Q7  How is this tested — what input proves it works, what input proves it fails?
```

A function whose author cannot answer all seven is not finished. The VERIFY
phase treats unanswerable functions as defects.

### §6.6 Failure-Mode-First Build Order

For EVERY function, build in this order. Not the happy path first — the failure
modes first.

```
1. CONTRACT      Define the signature, inputs, outputs, error types.
2. VALIDATE      Guard every input at the boundary (P2/P6/P8).
3. HANDLE ERRORS Write every catch/throw/recover path before the happy path.
4. HAPPY PATH    Implement the core logic last.
5. CLEANUP       Ensure every resource is released in finally (P4).
6. VERIFY        Confirm output contract holds in ALL paths (P10).
```

This order is binding. A build agent that writes the happy path first and bolts
on error handling afterward has violated the build order and its output will be
flagged by the R4 (Error Handling) audit layer.

## SECTION 7 — WHAT THIS SPEC EXPLICITLY BANS

The following patterns are **BANNED**. Each was tried in v4.4.1-rc or v4.4.2.
Each failed. Each is listed here so no build agent repeats the mistake.
Implementing any banned pattern is a spec violation and will be rejected at
VERIFY.

### §7.1 NO god-loop-hook.ts Replacing GodLoopOrchestrator

```
WHAT IT IS:
  A hook file that runs the entire 10-phase state machine inside the hook
  handler, making the hook the loop driver instead of the orchestrator class.

WHY IT WAS TRIED:
  The theory was that a hook could enforce the loop "more reliably" than a
  class, because hooks fire on every tool call regardless of model cooperation.

WHY IT FAILED:
  The hook fired on EVERY tool.execute.after, including tools unrelated to the
  loop. It re-evaluated state on every call. It injected dynamic content into
  the system prompt (banned pattern §7.2). It duplicated logic that already
  lived in the orchestrator. The two drifted out of sync. The hook became 430
  lines of unmaintainable, cache-destroying spaghetti.

WHAT TO DO INSTEAD:
  Keep GodLoopOrchestrator as the loop driver. The hook (poseidon-enforcer-
  hook.ts) is a thin guardrail (~120 lines) that only checks phase-to-tool
  alignment and escalates. The orchestrator owns state; the hook owns
  derailment detection.
```

### §7.2 NO Dynamic Content in the System Prompt

```
WHAT IT IS:
  Injecting phase/score/instruction text into the system prompt on every turn,
  so the system prompt changes turn-to-turn.

WHY IT WAS TRIED:
  To "keep the model aware" of the current phase without relying on the model
  re-reading tool output.

WHY IT FAILED:
  The runtime caches the system prompt's KV representation. Dynamic injection
  invalidates the cache every turn. Token cost exploded. Latency exploded.
  The model also got confused by a constantly-shifting system prompt.

WHAT TO DO INSTEAD:
  System prompt is STATIC (one line: "Poseidon Mode active. Call trident-
  poseidon to advance."). Dynamic phase detail goes in the MESSAGES channel
  via tool output. Cache is preserved.
```

### §7.3 NO Replacing extractScore() with a Filesystem-Diff Heuristic

```
WHAT IT IS:
  Replacing the proven, deterministic extractScore() (which compares pre/post
  finding signatures) with a heuristic that estimates score from how many
  files changed on disk.

WHY IT WAS TRIED:
  Filesystem diffs are cheap to compute; finding-signature comparison requires
  running the audit tool.

WHY IT FAILED:
  A file changing is not a defect being fixed. A build agent can rewrite a
  file 10 times and introduce 5 new defects. The heuristic reported "progress"
  (files changed) while the codebase got worse. Convergence broke. Score
  became an opinion.

WHAT TO DO INSTEAD:
  extractScore() compares preAuditFindings to postAuditFindings via SHA-256
  signatures. A finding is "resolved" only if its signature disappears from
  the post-audit. This is deterministic and cannot be gamed by file churn.
```

### §7.4 NO Mechanical Gates Replacing Tool-Based Instruction Flow

```
WHAT IT IS:
  Replacing the explicit, phase-specific text instructions returned by
  trident-poseidon with opaque mechanical gates that emit "you are derailed"
  or "phase blocked" with no actionable content.

WHY IT WAS TRIED:
  Mechanical gates seemed "safer" — they don't depend on the model reading
  instructions.

WHY IT FAILED:
  A gate that says "blocked" without telling the model WHAT to call leaves
  the model to guess. The model guessed wrong. The loop stalled on opaque
  signals with no path forward.

WHAT TO DO INSTEAD:
  Every phase returns EXPLICIT text instructions naming the exact tool to
  call next and the exact arguments. The model reads and executes. Gates
  exist only as a guardrail (the enforcer hook), not as the instruction
  channel.
```

### §7.5 NO Shark v4.9.9 Cargo-Cult Patterns

```
WHAT IT IS:
  Copying patterns from an unrelated system ("Shark v4.9.9") without
  understanding whether they apply to Trident's architecture.

WHY IT WAS TRIED:
  The patterns "worked" in Shark, so they were assumed to work here.

WHY IT FAILED:
  Shark's architecture is fundamentally different (different loop model,
  different state representation, different enforcement model). The copied
  patterns conflicted with Trident's tool-first architecture and introduced
  inconsistencies.

WHAT TO DO INSTEAD:
  Every pattern must be justified against THIS spec's three-layer
  architecture and the tool-first execution principle. No cargo-culting.
```

### §7.6 NO 4 New Helper Files Adding No Functional Value

```
WHAT IT IS:
  Creating activation-parser.ts, wave-manifest.ts, score-snapshot-scorer.ts,
  and similar helper files that duplicate logic already in the orchestrator
  or the tools.

WHY IT WAS TRIED:
  "Modularity." Splitting logic into small files seemed cleaner.

WHY IT FAILED:
  The helpers duplicated orchestrator logic. The two copies drifted. Bugs
  appeared in one copy but not the other. The files added indirection
  without adding function.

WHAT TO DO INSTEAD:
  Logic lives in the orchestrator or in the tools. New files are created
  only when they encapsulate a genuinely distinct responsibility that does
  not overlap with existing code.
```

### §7.7 NO 430-Line Hook State Machine

```
WHAT IT IS:
  A hook file that grew to ~430 lines because it accumulated state-machine
  logic, prompt-building logic, scoring heuristics, and escalation logic.

WHY IT WAS TRIED:
  Incremental feature addition without a size budget.

WHY IT FAILED:
  A 430-line hook is unmaintainable, untestable, and violates the
  "hook is a guardrail" principle. It also nearly always ends up injecting
  dynamic content (banned §7.2).

WHAT TO DO INSTEAD:
  poseidon-enforcer-hook.ts is capped at ~120 lines (hard ceiling 150). It
  does exactly: read state.json, check phase-to-tool alignment, escalate.
  Nothing else.
```

### §7.8 NO Empty Handler No-Ops

```
WHAT IT IS:
  catch blocks that do nothing (catch {}), functions that return undefined
  implicitly, handlers that swallow events.

WHY IT WAS TRIED:
  To "get past" an error quickly during development.

WHY IT FAILED:
  P3 violation. Silent failures corrupt state invisibly. The loop spins on
  broken state with no diagnostic. This is the worst failure mode.

WHAT TO DO INSTEAD:
  Every catch logs (with context) and either recovers or propagates. Every
  function returns an explicit value in every path (P10).
```

### §7.9 NO Problem-Solving Skipped When Stalled

```
WHAT IT IS:
  When the score stalls, continuing to dispatch the same kind of wave instead
  of invoking trident-problem-solving to diagnose the stall.

WHY IT WAS TRIED:
  "Just try harder" — dispatching more waves felt like progress.

WHY IT FAILED:
  If the score is stalled, the current approach is not working. Dispatching
  more of the same burns cycles without convergence. cycle > 50 → FAILED.

WHAT TO DO INSTEAD:
  After 3 stalled cycles, DECIDE transitions to PROBLEM_SOLVE. The 6-layer
  diagnosis identifies why progress stopped. PLAN then uses the revised
  approach.
```

### §7.10 NO ScoreSnapshotScorer Replacing Proven Regex Scoring

```
WHAT IT IS:
  A "smarter" scoring class that replaces extractScore() with a weighted,
  multi-factor heuristic.

WHY IT WAS TRIED:
  extractScore() seemed "too simple" (regex + set comparison).

WHY IT FAILED:
  The simple scorer is deterministic and cannot be gamed. The "smarter"
  scorer introduced tunable weights that became opinions. Different runs
  produced different scores for the same codebase (R1 violation).

WHAT TO DO INSTEAD:
  extractScore() stays. It compares finding-signature sets. Simple,
  deterministic, reproducible.
```

### §7.11 NO messages.transform as Primary Instruction Channel

```
WHAT IT IS:
  Using the messages.transform hook to inject the primary phase instructions,
  so instructions arrive via message transformation rather than tool output.

WHY IT WAS TRIED:
  messages.transform runs on every turn and can inject text reliably.

WHY IT FAILED:
  Instructions arriving via transform bypass the tool-first principle. The
  model cannot distinguish tool-computed instructions from transform-injected
  text. The instruction provenance is lost. Debugging becomes impossible.

WHAT TO DO INSTEAD:
  Instructions come from trident-poseidon tool output (the MESSAGES channel
  via tool result, not via messages.transform). Provenance is explicit.
```

### §7.12 NO activation-parser.ts or wave-manifest.ts as New Files

```
WHAT IT IS:
  Standalone files that parse "activation" state or represent wave manifests
  outside the orchestrator.

WHY IT WAS TRIED:
  To decouple parsing/representation from the orchestrator.

WHY IT FAILED:
  Activation state IS orchestrator state. Wave manifests ARE orchestrator

WHAT TO DO INSTEAD:
  Activation is a field in GodLoopState. Wave manifests are the
  WaveManifest interface produced by phasePlan() inside the orchestrator.

```

### 8.3 GodLoopPhase — 10-Phase Machine

```typescript
type GodLoopPhase =
  | 'INIT'       // Initialize state, read findings baseline
  | 'AUDIT'      // Run trident-code-audit, compute score
  | 'PLAN'       // Generate build spec from remaining findings
  | 'DISPATCH'   // Dispatch build agents in parallel waves
  | 'VERIFY'     // Verify evidence gate + content hash diff
  | 'COLLECT'    // Synthesize context bridge for next wave
  | 'GAP'        // Analyze gaps, detect stall, problem-solve
  | 'SOLVE'      // Execute revised approach from problem-solving
  | 'LOCK'       // Score >= 96 and gate passed — lock cycle
  | 'FAILED';    // Max cycles reached or unrecoverable error
```

Phase transitions are mechanical and deterministic:

```
INIT -> AUDIT -> PLAN -> DISPATCH -> VERIFY -> COLLECT -> GAP
                                                        |
                                    +-------------------+
                                    |                   |
                               stalled:            improving:
                               SOLVE              AUDIT (next cycle)
                                    |
                               SOLVE -> DISPATCH -> VERIFY -> ...
                                                        |
                                              score >= 96
                                              gate passed
                                                        |
                                                        v
                                                     LOCK
```

### 8.4 PhaseResult Interface

### 8.5 runPhase() — Main Entry Point

This is the **only** method that the `trident-poseidon` tool calls. It is the single funnel through which all phases flow.

```typescript
/**
 * Main entry point for Poseidon Mode.
 * Called by trident-poseidon tool with action="start" or action="status".
 *
 * Contract:
 *   1. Read state.json from disk (P1: verify file exists before use)
 *   2. Switch on current phase
 *   3. Execute the mechanical action for that phase
 *   4. Write updated state.json atomically (P5: snapshot-and-rollback)
 *   5. Write evidence to Merkle chain
 *   6. Return PhaseResult with explicit text instructions
 *
 * Error handling (P3): every catch block logs the error,
 *   writes FAILED state if unrecoverable, or recovers with
 *   a degraded instruction set.
 */
async runPhase(targetPath: string, sessionId: string): Promise<PhaseResult> {
  // --- P1: DEFENSIVE READ ---
  const statePath = path.join(targetPath, '.trident', 'god-loop', 'state.json');
  let state: GodLoopState;
  try {
    const raw = fs.readFileSync(statePath, 'utf-8');
    state = JSON.parse(raw) as GodLoopState;
  } catch (err: unknown) {
    // State file missing or corrupt — initialize fresh
    if (err instanceof Error) {
      console.error(`[POSEIDON] Failed to read state: ${err.message}`);
    }
    state = this.initializeState(targetPath, sessionId);
    this.writeStateAtomic(statePath, state);
  }

  // --- P2: PHASE DISPATCH ---
  let result: PhaseResult;
  try {
    switch (state.phase) {
      case 'INIT':
        result = await this.phaseInit(state, targetPath);
        break;
      case 'AUDIT':
        result = await this.phaseAudit(state, targetPath);
        break;
      case 'PLAN':
        result = await this.phasePlan(state, targetPath);
        break;
      case 'DISPATCH':
        result = await this.phaseDispatch(state, targetPath);
        break;
      case 'VERIFY':
        result = await this.phaseVerify(state, targetPath);
        break;
      case 'COLLECT':
        result = await this.phaseCollect(state, targetPath);
        break;
      case 'GAP':
        result = await this.phaseGap(state, targetPath);
        break;
      case 'SOLVE':
        result = await this.phaseSolve(state, targetPath);
        break;
      case 'LOCK':
        result = await this.phaseLock(state, targetPath);
        break;
      case 'FAILED':
        result = await this.phaseFailed(state, targetPath);
        break;
      default:
        // P8: Config validation — unknown phase is unrecoverable
        state.phase = 'FAILED';
        state.error = `Unknown phase: ${state.phase}`;
        this.writeStateAtomic(statePath, state);
        result = this.buildResult(
          state, 'FAILED',
          `UNRECOVERABLE: Unknown phase. State marked as FAILED.`,
          [],
        );
    }
  } catch (err: unknown) {
    // P3: ERROR COMPLETENESS — log, write FAILED state, return error result
    const msg = err instanceof Error ? err.message : String(err);
    console.error(`[POSEIDON] Phase ${state.phase} failed: ${msg}`);
    state.phase = 'FAILED';
    state.error = msg;
    state.failedAt = new Date().toISOString();
    this.writeStateAtomic(statePath, state);
    result = this.buildResult(
      state, 'FAILED',
      `PHASE ${state.phase.toUpperCase()} CRASHED: ${msg}. State written as FAILED. Check logs.`,
      [`Phase crash: ${msg}`],
    );
  }

  // --- P4: STATE + EVIDENCE WRITES ---
  // State was already written inside the phase method.
  // Write evidence to Merkle chain.
  try {
    await this.evidenceStore.append({
      phase: state.phase,
      cycle: state.cycle,
      score: result.score,
      timestamp: new Date().toISOString(),
      action: `Phase ${result.phase} to ${result.nextPhase}`,
    });
  } catch (err: unknown) {
    const msg = err instanceof Error ? err.message : String(err);
    result.warnings.push(`Evidence write failed: ${msg}`);
    console.warn(`[POSEIDON] Evidence write failed: ${msg}`);
  }

  return result;
}
```

### 8.6 Enhanced extractScore() — Hash-Based Progressive Scoring

The existing `extractScore()` reads score from audit output text via regex. This is **preserved** as a sanity check. A new `computeScore()` method (Section 9) produces the **authoritative** hash-based score.

```typescript
/**
 * ENHANCED extractScore — now delegates to computeScore() for the
 * authoritative score, while keeping the regex-based score as a
 * sanity check. If they diverge by more than 20 points, a warning
 * is logged.
 *
 * OLD behavior: read "Score: X/100" from audit output text
 * NEW behavior: compute hash-based score, cross-check with regex
 *
 * @param auditOutput  Raw text from trident-code-audit
 * @param preAudit     Finding signatures from baseline audit
 * @param postAudit    Finding signatures from current audit
 * @returns            { score: number, divergence: number | null }
 */
extractScore(
  auditOutput: string,
  preAudit: FindingSignature[],
  postAudit: FindingSignature[],
): { score: number; divergence: number | null } {
  // --- Regex score (sanity check) ---
  // P2: Type certainty — parse with fallback
  let regexScore = -1;
  const match = auditOutput.match(/Score:\s*(\d+)\s*\/\s*100/);
  if (match && match[1]) {
    const parsed = parseInt(match[1], 10);
    if (!Number.isNaN(parsed) && parsed >= 0 && parsed <= 100) {
      regexScore = parsed;
    }
  }

  // --- Hash-based score (authoritative) ---
  const hashResult = this.computeScore(preAudit, postAudit);
  const authoritativeScore = hashResult.score;

  // --- Divergence check ---
  let divergence: number | null = null;
  if (regexScore >= 0) {
    divergence = Math.abs(authoritativeScore - regexScore);
    if (divergence > 20) {
      console.warn(
        `[POSEIDON] SCORE DIVERGENCE: hash=${authoritativeScore}, regex=${regexScore}, ` +
        `delta=${divergence}. Using hash-based score as authoritative.`,
      );
    }
  }

  return { score: authoritativeScore, divergence };
}
```

### 8.7 Enhanced verifyCycle() — Evidence Gate + Content Hash Diff

The existing `verifyCycle()` checks if the cycle should advance. It is **enhanced** to call `evidenceStore.verifyAndGate()` (Bug Fix #3 — this method existed but was never called).

```typescript
/**
 * ENHANCED verifyCycle — now performs three checks:
 *   1. Merkle chain integrity (calls verifyAndGate — bug fix #3)
 *   2. Content hash diff on all modified files
 *   3. passRate >= 0.96 threshold
 *
 * OLD behavior: checked passRate from a field, no chain verification
 * NEW behavior: full gate verification with tamper detection
 *
 * @param state   Current GodLoopState
 * @returns       { advance: boolean, reason: string, details: EvidenceGateDetails }
 */
async verifyCycle(
  state: GodLoopState,
): Promise<{ advance: boolean; reason: string; details: EvidenceGateDetails | null }> {

  // --- CHECK 1: Merkle chain integrity ---
  const gateResult = await this.evidenceStore.verifyAndGate();

  if (!gateResult.passed) {
    // Evidence chain is broken or incomplete.
    // Severity determines recovery action.
    if (gateResult.passRate < 0.50) {
      // Severe corruption — mark FAILED, cannot trust any evidence
      state.phase = 'FAILED';
      state.error = `Severe evidence corruption: passRate=${gateResult.passRate}`;
      this.writeStateAtomic(state.statePath, state);
      return {
        advance: false,
        reason: `SEVERE CORRUPTION: Evidence gate failed at passRate=${gateResult.passRate}. ` +
                `Broken at entry ${gateResult.details.firstBrokenEntry}. Cycle FAILED.`,
        details: gateResult.details,
      };
    }
    // Partial corruption (0.50-0.95) — re-dispatch the wave
    return {
      advance: false,
      reason: `EVIDENCE GATE FAILED. Re-dispatching wave ${state.wave}. ` +
              `Pass rate: ${gateResult.passRate}. Broken at entry ${gateResult.details.firstBrokenEntry}.`,
      details: gateResult.details,
    };
  }

  // --- CHECK 2: Content hash diff on modified files ---
  const modifiedFiles = state.modifiedFiles ?? [];
  let hashMismatches = 0;

  for (const filePath of modifiedFiles) {
    // P1: Defensive — verify file exists
    if (!fs.existsSync(filePath)) {
      console.warn(`[POSEIDON] Modified file no longer exists: ${filePath}`);
      hashMismatches++;
      continue;
    }

    const currentHash = this.hashFile(filePath);
    const recordedHash = state.fileHashes.get(filePath);

    if (recordedHash && currentHash !== recordedHash) {
      // File changed since dispatch — expected (builder modified it)
      state.fileHashes.set(filePath, currentHash);
    } else if (!recordedHash) {
      // New file — record its hash
      state.fileHashes.set(filePath, currentHash);
    }
  }

  if (hashMismatches > modifiedFiles.length * 0.5) {
    return {
      advance: false,
      reason: `More than 50% of modified files are missing. Re-dispatching wave ${state.wave}.`,
      details: gateResult.details,
    };
  }

  // --- CHECK 3: passRate threshold ---
  // passRate comes from the score engine (closed + modified * 0.5 / total)
  if (state.passRate < 0.96) {
    return {
      advance: false,
      reason: `Pass rate ${state.passRate.toFixed(4)} < 0.96 threshold. ` +
              `Score: ${state.score}/100. Continuing to next cycle.`,
      details: gateResult.details,
    };
  }

  // All checks passed — cycle can advance or lock
  return {
    advance: true,
    reason: `All gates passed. passRate=${state.passRate.toFixed(4)}, ` +
            `chain intact, ${modifiedFiles.length} files verified.`,
    details: gateResult.details,
  };
}
```

### 8.8 detectStall() — Score Stagnation Detection

Stall detection prevents infinite loops where the builder makes changes that do not improve the score. A stall is defined as no score improvement for 3 consecutive cycles.

```typescript
/**
 * Detect score stagnation using CycleTracker history.
 *
 * Stall = no score improvement for STALL_THRESHOLD (3) consecutive cycles.
 * "Improvement" means the score increased by at least STALL_MIN_GAIN (1 point).
 * A score that stays flat or decreases counts as no improvement.
 *
 * @returns  StallAnalysis with cyclesSinceImprovement and diagnostic text
 */
detectStall(): StallAnalysis {
  const STALL_THRESHOLD = 3;
  const STALL_MIN_GAIN = 1; // points

  const history = this.cycleTracker.getHistory();

  if (history.length < STALL_THRESHOLD + 1) {
    return {
      stalled: false,
      cyclesSinceImprovement: history.length,
      analysis: `Insufficient history (${history.length} cycles). ` +
                `Need ${STALL_THRESHOLD + 1} to detect stall.`,
    };
  }

  let cyclesSinceImprovement = 0;
  const recent = history.slice(-(STALL_THRESHOLD + 1));

  // Walk backward from most recent, counting cycles without improvement
  for (let i = recent.length - 1; i > 0; i--) {
    const current = recent[i].score;
    const previous = recent[i - 1].score;
    const gain = current - previous;

    if (gain >= STALL_MIN_GAIN) {
      break; // Found improvement — stop counting
    }
    cyclesSinceImprovement++;
  }

  const stalled = cyclesSinceImprovement >= STALL_THRESHOLD;

  const analysis = stalled
    ? `STALLED: Score has not improved by >= ${STALL_MIN_GAIN} point for ` +
      `${cyclesSinceImprovement} consecutive cycles. ` +
      `Score history: [${recent.map(c => c.score).join(', ')}]. ` +
      `Problem-solving is REQUIRED before continuing.`
    : `Progressing: ${cyclesSinceImprovement} cycles since last improvement ` +
      `(threshold: ${STALL_THRESHOLD}). Score history: ` +
      `[${recent.map(c => c.score).join(', ')}].`;

  return { stalled, cyclesSinceImprovement, analysis };
}
```

### 8.9 runProblemSolve() — Invoke 6-Layer Mental Framework

When `detectStall()` returns `stalled: true`, the GAP phase calls `runProblemSolve()`. This invokes `trident-problem-solving` with a pre-built mental framework — it **never guesses**.

```typescript
/**
 * Invoke trident-problem-solving when score has stalled.
 *
 * Constructs a pre-built mental framework from:
 *   - Cycle history (what scores were achieved)
 *   - Persistent findings (what refuses to be fixed)
 *   - Failed approaches (what the builder already tried)
 *
 * The 6-layer pipeline (assumption -> action -> observe -> gap -> meta -> verify)
 * produces a revised working plan that the SOLVE phase executes.
 *
 * @param state     Current GodLoopState
 * @param stall     StallAnalysis from detectStall()
 * @returns         Revised approach with specific action items
 */
async runProblemSolve(
  state: GodLoopState,
  stall: StallAnalysis,
): Promise<ProblemSolveResult> {

  // Gather persistent findings (appeared in all recent cycles with same hash)
  const persistentFindings = this.findingTracker.getPersistentFindings();

  // Gather failed approaches (findings that were dispatched but not resolved)
  const failedApproaches = this.findingTracker.getFailedApproaches();

  // Build the problem description
  const problem = `Score has stalled at ${state.score}/100 for ` +
    `${stall.cyclesSinceImprovement} cycles. ` +
    `${persistentFindings.length} findings are persistent (unchanged hash across cycles). ` +
    `${failedApproaches.length} approaches have been tried without resolution.`;

  // Build the reasoning chain (evidence-based, no guessing)
  const reasoning: string[] = [
    `Cycle history: ${this.cycleTracker.getHistory().map(c => `C${c.cycle}=${c.score}`).join(', ')}`,
    `Persistent findings: ${persistentFindings.slice(0, 10).map(f => `${f.id} (${f.issue})`).join(', ')}`,
    `Failed approaches: ${failedApproaches.slice(0, 5).map(a => `${a.findingId}: ${a.approach}`).join(', ')}`,
    `Pattern analysis: ${persistentFindings.filter(f => f.issue === 'THEATRICAL_CODE').length} ` +
      `theatrical, ${persistentFindings.filter(f => f.severity === 'critical').length} critical`,
    `Architecture constraints: ${state.architectureConstraints?.join('; ') ?? 'none identified'}`,
  ];

  // Build the working plan template (problem-solving will refine this)
  const workingPlan: string[] = [
    `Re-audit files containing persistent findings with trident-code-audit for fresh signatures`,
    `Group persistent findings by root cause (not by file)`,
    `For each root cause group, design a single fix that addresses ALL findings in the group`,
    `If a finding has been persistent for 5+ cycles, consider it architectural — ` +
      `document as known-issue and exclude from score`,
    `Re-dispatch with revised build spec targeting root causes, not symptoms`,
  ];

  // Call trident-problem-solving tool (via task subagent)
  const solveResult = await this.invokeProblemSolving({
    targetPath: state.targetPath,
    problem,
    reasoning,
    workingPlan,
  });

  return {
    revisedApproach: solveResult.workingPlan,
    rootCauseAnalysis: solveResult.rootCauseAnalysis,
    keyInsights: solveResult.insights,
    newFindings: solveResult.newFindings ?? [],
    shouldExclude: solveResult.shouldExclude ?? [],
  };
}
```

The return type:

```typescript

  /** Revised working plan from the 6-layer pipeline */
  revisedApproach: string[];
  /** Root cause analysis (the "why" behind the stall) */
  rootCauseAnalysis: string;
  /** Key insights discovered during problem-solving */
  keyInsights: string[];
  /** New findings discovered during analysis */
  newFindings: FindingSignature[];
  /** Findings that should be excluded (architectural, 5+ cycles persistent) */
  shouldExclude: string[];
}
```

#### 8.10.2 phaseAudit()

```typescript
/**
 * AUDIT phase: Run full audit, compute progressive score.
 *
 * Calls: trident-code-audit (full mode) + computeScore()
 * Writes: state.json with updated findings, score, passRate
 * Returns: instructions to proceed to PLAN or LOCK
 */
private async phaseAudit(state: GodLoopState, targetPath: string): Promise<PhaseResult> {
  // 2. Call trident-code-audit full mode
  const fullAudit = await this.runCodeAudit(targetPath, 'full');

  // 3. Parameters: targetPath, mode='full'
  // 4. Process: extract finding signatures for current cycle
  const currentFindings = this.extractFindingSignatures(fullAudit, state.cycle);

  // Compute hash-based score
  const scoreResult = this.computeScore(
    state.baselineFindings,
    currentFindings,
  );

  // Cross-check with regex (sanity)
  const { score, divergence } = this.extractScore(
    fullAudit.outputText,
    state.baselineFindings,
    currentFindings,
  );

  // 5. State writes
  state.score = score;
  state.currentFindings = currentFindings;
  state.passRate = scoreResult.closed / Math.max(scoreResult.total, 1);
  state.lastAuditAt = new Date().toISOString();

  // Update finding tracker
  this.findingTracker.update(currentFindings, state.cycle);

  // Record score in cycle tracker
  this.cycleTracker.record(state.cycle, score);

  // Determine next phase
  const warnings: string[] = [];
  if (divergence !== null && divergence > 20) {
    warnings.push(`Score divergence: hash=${score}, regex delta=${divergence}`);
  }

  let nextPhase: GodLoopPhase;
  let instructions: string;

  if (score >= 96 && state.passRate >= 0.96) {
    // Score threshold met — advance to VERIFY for gate check
    state.phase = 'VERIFY';
    nextPhase = 'VERIFY';
    instructions =
      `## AUDIT COMPLETE — THRESHOLD MET\n\n` +
      `Score: ${score}/100\n` +
      `Closed: ${scoreResult.closed}, Modified: ${scoreResult.modified}, ` +
      `Persistent: ${scoreResult.persistent}, Total: ${scoreResult.total}\n` +
      `Pass rate: ${state.passRate.toFixed(4)}\n\n` +
      `## NEXT ACTION\n` +
      `Call trident-poseidon action="start" targetPath="${targetPath}"\n` +
      `Phase advances to VERIFY for evidence gate check.`;
  } else {
    // Continue to PLAN
    state.phase = 'PLAN';
    nextPhase = 'PLAN';
    instructions =
      `## AUDIT COMPLETE\n\n` +
      `Score: ${score}/100\n` +
      `Closed: ${scoreResult.closed}, Modified: ${scoreResult.modified}, ` +
      `Persistent: ${scoreResult.persistent}, Total: ${scoreResult.total}\n` +
      `Pass rate: ${state.passRate.toFixed(4)}\n\n` +
      `## NEXT ACTION\n` +
      `Call trident-poseidon action="start" targetPath="${targetPath}"\n` +
      `Phase advances to PLAN.`;

  this.writeStateAtomic(state.statePath, state);
  return this.buildResult(state, nextPhase, instructions, warnings);
```

#### 8.10.4 phaseDispatch()

```typescript
/**
 * DISPATCH phase: Emit explicit instructions for parallel agent dispatch.
 *
 * NOTE: The orchestrator does NOT dispatch agents itself.
 * It writes instructions that the model reads and executes.
 * The model calls task() for each agent in ONE message.
 *
 * Calls: reads wave manifest from disk
 * Writes: state.json with dispatched agent IDs
 * Returns: instructions with full agent dispatch text
 */
private async phaseDispatch(state: GodLoopState, targetPath: string): Promise<PhaseResult> {
  // 2. Read wave manifest
  // P1: Defensive — verify manifest exists
  const manifestPath = state.waveManifestPath;
  if (!manifestPath || !fs.existsSync(manifestPath)) {
    state.phase = 'FAILED';
    state.error = 'Wave manifest missing during DISPATCH';
    this.writeStateAtomic(state.statePath, state);
    return this.buildResult(
      state, 'FAILED',
      `UNRECOVERABLE: Wave manifest not found at ${manifestPath}.`,
      ['Missing wave manifest'],
    );
  }

  const manifest: WaveManifest = JSON.parse(
    fs.readFileSync(manifestPath, 'utf-8'),
  );

  // 3. Parameters: manifest is read directly
  // 4. Process: build dispatch instructions for each agent

  // 5. State writes
  state.phase = 'VERIFY';
  state.dispatchedAgents = manifest.agents.map(a => a.id);
  state.dispatchedAt = new Date().toISOString();
  this.writeStateAtomic(state.statePath, state);

  // 6. Text instruction — the MODEL dispatches agents
  const agentBlocks = manifest.agents.map(agent => {
    return `### Agent ${agent.id}: ${agent.module}\n` +
      `- **File:** ${agent.filePath}\n` +
      `- **Task:** ${agent.spec}\n` +
      `- **Findings to address:** ${agent.findings.length}\n` +
      `- **Allowed tools:** ${agent.allowedTools.join(', ')}\n` +
      `- **Bash access:** ${agent.bashAccess ? 'YES' : 'NO'}\n` +
      `- **Context bridge:** Read T1 at ${manifest.t1ContextPath}`;
  }).join('\n\n');

  const instructions =
    `## WAVE ${state.wave} DISPATCH\n\n` +
    `Dispatch ${manifest.agents.length} agents in parallel by calling ` +
    `task() ${manifest.agents.length} times in ONE message.\n\n` +
    agentBlocks + '\n\n' +
    `## AFTER ALL AGENTS RETURN\n` +
    `Call trident-poseidon action="start" targetPath="${targetPath}"\n` +
    `Phase will advance to VERIFY for evidence gate check.`;

  return this.buildResult(state, 'VERIFY', instructions, []);
}
```

#### 8.10.5 phaseVerify()

```typescript

 * VERIFY phase: Run evidence gate verification.
 *
 * Calls: verifyCycle() (enhanced with verifyAndGate)

private async phaseVerify(state: GodLoopState, targetPath: string): Promise<PhaseResult> {
  // 2. Call enhanced verifyCycle
  const gateResult = await this.verifyCycle(state);

  if (!gateResult.advance) {
    // Gate failed — either re-dispatch or FAIL

      `Call trident-poseidon action="start" targetPath="${targetPath}"\n` +
      `Phase returns to DISPATCH for wave ${state.wave}.`,
      [],

  // Gate passed — check if score threshold met
  if (state.score >= 96 && state.passRate >= 0.96) {
    state.phase = 'LOCK';
    this.writeStateAtomic(state.statePath, state);
    return this.buildResult(
      state, 'LOCK',

        `## NEXT ACTION\n` +
        `Call trident-poseidon action="start" targetPath="${targetPath}"\n` +
        `Phase advances to LOCK.`,
      [],
    );

  // Gate passed but score < 96 — collect context for next wave
  state.phase = 'COLLECT';
  this.writeStateAtomic(state.statePath, state);
  return this.buildResult(
    state, 'COLLECT',
    `## VERIFY PASSED\n\n${gateResult.reason}\n\n` +
      `## NEXT ACTION\n` +
      `Call trident-poseidon action="start" targetPath="${targetPath}"\n` +
```

## SECTION 9: SCORE ENGINE

### 9.1 Overview

The Score Engine replaces the broken "count remaining findings" approach with a **hash-based progressive scoring** system. The old approach was fundamentally flawed: with 868 findings (many in bundle noise), fixing 100 left the score at 0 because the denominator never changed. The new approach tracks **which specific findings were actually addressed** by comparing source code hashes before and after each build wave.

### 9.2 FindingSignature Interface

Every finding tracked by the Score Engine is represented as a `FindingSignature`. This is the atomic unit of scoring.

```typescript
interface FindingSignature {
  /** Unique finding ID (generated from file + line + issue type) */
  id: string;
  /** Source file path (relative to target) */
  file: string;
  /** Line number where the finding was detected */
  line: number;
  /** Issue type (e.g., 'EMPTY_CATCH', 'THEATRICAL_CODE', 'MISSING_RETURN') */
  issue: string;
  /** Severity: 'critical' | 'high' | 'medium' | 'low' */
  severity: string;
  /** Confidence score 0.0-1.0 from the audit engine */
  confidence: number;
  /**
   * SHA-256 hash of surrounding code block (+/- 5 lines from the finding).

  sourceHash: string;

  /** Lifecycle status */
  status: 'pending' | 'modified' | 'closed' | 'persistent';
  /** First cycle this finding was observed */
  firstSeenAt: number;
  /** Most recent cycle this finding was observed */
  lastSeenAt: number;
  /** Whether a fix was dispatched to a build agent */
  fixAttempted: boolean;
  /** Whether the fix was verified by post-audit */
  fixVerified: boolean;
```

### 9.3 computeScore() Method

```typescript
/**
 * Compute hash-based progressive score.
 *
 * For each pre-audit finding, checks if it still exists in post-audit:
 *   - sourceHash changed -> 'modified' (builder touched it)
 *   - finding gone -> 'closed' (resolved)
 *   - same sourceHash -> 'persistent' (unfixed)
 *
 * Progressive score = (closed + modified * 0.5) / totalSourceFindings * 100
 *
 * @param preAudit   Finding signatures from baseline audit
 * @param postAudit  Finding signatures from current audit
 * @returns          Score breakdown with closed/modified/persistent counts
 */
function computeScore(
  preAudit: FindingSignature[],
  postAudit: FindingSignature[],
): { score: number; closed: number; modified: number; persistent: number; total: number } {

  // Index post-audit findings by ID for O(1) lookup
  const postMap = new Map<string, FindingSignature>();
  for (const finding of postAudit) {
    postMap.set(finding.id, finding);
  }

  let closed = 0;      // Finding resolved (not in post-audit)
  let modified = 0;    // Finding touched (hash changed)
  let persistent = 0;  // Finding unfixed (same hash)
  let totalSourceFindings = 0;

  for (const preFinding of preAudit) {
    // --- Confidence floor: exclude low-confidence findings ---
    if (preFinding.confidence < 0.85) {
      continue; // Not counted in score
    }

    // --- Type weighting: source findings only in denominator ---
    if (preFinding.fileType === 'bundle') {
      continue; // Bundle findings do not count in denominator
    }
    // fileType === 'source' -> counts
    totalSourceFindings++;

    const postFinding = postMap.get(preFinding.id);

    if (!postFinding) {
      // Finding no longer exists — resolved
      closed++;
      preFinding.status = 'closed';
      preFinding.fixVerified = true;
    } else if (postFinding.sourceHash !== preFinding.sourceHash) {
      // Finding still exists but code changed — builder modified it
      modified++;
      preFinding.status = 'modified';
      preFinding.sourceHash = postFinding.sourceHash;
      preFinding.lastSeenAt = postFinding.lastSeenAt;
    } else {
      // Finding exists with identical hash — builder did not touch it
      persistent++;
      preFinding.status = 'persistent';
      preFinding.lastSeenAt = postFinding.lastSeenAt;
    }
  }

  // --- Progressive score ---
  // (closed findings count fully, modified count half)
  const denominator = Math.max(totalSourceFindings, 1); // P2: prevent div-by-zero
  const rawScore = ((closed + modified * 0.5) / denominator) * 100;

  // Round to nearest integer
  const score = Math.round(rawScore);

  return { score, closed, modified, persistent, total: totalSourceFindings };
}
```

### 9.4 Hash Function for Finding Signatures

```typescript
import * as crypto from 'crypto';
import * as fs from 'fs';
import * as path from 'path';

/**
 * Hash the code context surrounding a finding.
 *
 * Takes the file path, line number, and +/- 5 lines of surrounding code,
 * concatenates them, and computes SHA-256.
 *
 * This hash is used to detect whether the builder ACTUALLY modified
 * the code near a finding:
 *   - Unchanged hash -> finding is 'persistent' (builder did not touch it)
 *   - Changed hash -> finding is 'modified' (builder made a change)
 *
 * @param file         Relative file path
 * @param line         Line number of the finding
 * @param codeContext  The +/- 5 lines of surrounding code (pre-extracted)
 * @returns            Hex-encoded SHA-256 hash
 */
function hashFindingContext(file: string, line: number, codeContext: string): string {
  // Concatenate: file path + line number + code context
  // This ensures that the same issue in different files or at different
  // lines produces different hashes.
  const input = `${file}:${line}:${codeContext}`;

  // SHA-256 hash
  return crypto.createHash('sha256').update(input, 'utf-8').digest('hex');
}

/**
 * Extract +/- 5 lines of code context from a file for hashing.
 *
 * @param filePath  Absolute path to the source file
 * @param line      Line number of the finding (1-indexed)
 * @returns         The code context string, or empty string on error
 */
function extractCodeContext(filePath: string, line: number): string {
  const CONTEXT_RADIUS = 5;

  try {
    // P1: Defensive — verify file exists
    if (!fs.existsSync(filePath)) {
      return '';
    }

    const content = fs.readFileSync(filePath, 'utf-8');
    const lines = content.split('\n');

    // P2: Type certainty — validate line number
    const startLine = Math.max(0, line - 1 - CONTEXT_RADIUS);
    const endLine = Math.min(lines.length, line + CONTEXT_RADIUS);

    return lines.slice(startLine, endLine).join('\n');
  } catch (err: unknown) {
    // P3: Error completeness — log and return empty
    const msg = err instanceof Error ? err.message : String(err);
    console.warn(`[SCORE_ENGINE] Failed to extract context from ${filePath}: ${msg}`);
    return '';
  }
}
```

### 9.5 Progressive Scoring: Old vs New

**OLD approach (broken):**

```
Score = 100 - (remainingFindings / totalFindings * 100)
```

Problems:
- With 868 total findings (many bundle noise), fixing 100 source findings
  leaves: `100 - (768 / 868 * 100) = 100 - 88 = 12/100`
- The score barely moves because bundle noise inflates the denominator.
- Builders see no progress -> motivation drops -> loop stalls.
- No distinction between "builder touched it" and "builder fixed it."

**NEW approach (progressive):**

```
Score = (closed + modified * 0.5) / totalSourceFindings * 100
```

Where:
- `closed` = findings that no longer appear in post-audit (resolved)
- `modified` = findings whose sourceHash changed (builder touched them)
- `totalSourceFindings` = high-confidence findings in `.ts` files only
- Bundle (`.js`) findings are excluded from the denominator

Results:
- Fixed 100 of 200 source findings -> score = 50
- Fixed 150 of 200 -> score = 75
- Fixed 190 of 200 -> score = 95
- Fixed 193 of 200 -> score = 96.5 ~ 97 -> LOCKED

The score **progresses** with each fix and does not stagnate due to bundle noise.

### 9.6 Type Weighting Table

| File Type | Extension | Weight | Rationale |
|---|---|---|---|
| Source | `.ts` | 1.0 | Real source code, findings matter, builder can fix |
| Bundle | `.js` (minified) | 0.1 | Generated code, findings are noise, builder cannot fix |
| Declaration | `.d.ts` | 0.0 | Type declarations only, not executable |
| Config | `.json` | 0.0 | Not executable code, no runtime impact |
| Map | `.js.map` | 0.0 | Source maps, debugging artifacts only |

The type is determined by file extension at signature creation time:

```typescript
function classifyFileType(filePath: string): 'source' | 'bundle' {
  if (filePath.endsWith('.ts') && !filePath.endsWith('.d.ts')) {
    return 'source';
  }
  if (filePath.endsWith('.d.ts')) {
    return 'bundle'; // Treated as 0.0 weight, excluded from score
  }
  if (filePath.endsWith('.js') || filePath.endsWith('.mjs')) {
    // Check if minified (single-line, >500 chars per line average)
    return 'bundle';
  }
  return 'bundle'; // Default: anything else is non-source
}
```

### 9.7 Confidence Floor Specification

Not all findings are equal. Low-confidence findings from the audit engine are often false positives. The confidence floor prevents them from polluting the score:

| Confidence Range | Action | Score Impact |
|---|---|---|
| `< 0.50` | **EXCLUDED** entirely | Not counted in any calculation |
| `0.50 - 0.84` | **LOGGED** but not dispatched | Appears in audit report, does not affect score |
| `>= 0.85` | **DISPATCHED** to build agents | Counts in score denominator and numerator |

The audit engine still **reports** all findings (for completeness), but only high-confidence findings drive the God Loop. This prevents the loop from chasing phantom issues that cannot be verified.

```typescript
/**
 * Apply confidence floor to findings.
 * Returns only findings that meet the dispatch threshold.
 *
 * @param findings  All findings from the audit
 * @returns         High-confidence findings (>= 0.85)
 */
function applyConfidenceFloor(findings: FindingSignature[]): FindingSignature[] {
  return findings.filter(f => {
    if (f.confidence < 0.50) {
      // Excluded entirely — not even logged for tracking
      return false;
    }
    if (f.confidence < 0.85) {
      // Logged but not dispatched
      console.debug(
        `[SCORE_ENGINE] Low-confidence finding logged but excluded: ` +
        `${f.id} (${f.issue}, confidence=${f.confidence.toFixed(2)})`,
      );
      return false;
    }
    return true; // confidence >= 0.85 — dispatched
  });
}
```

### 9.8 FindingTracker — Lifecycle Management

The `FindingTracker` class manages the lifecycle of findings across cycles:

```typescript
class FindingTracker {
  private findings: Map<string, FindingSignature> = new Map();
  private excludedIds: Set<string> = new Set();
  private failedApproaches: Map<string, Array<{ cycle: number; approach: string }>> = new Map();

  /**
   * Update findings from the latest audit.
   * Tracks which findings appeared, changed, or disappeared.
   */
  update(currentFindings: FindingSignature[], cycle: number): void {
    const currentIds = new Set(currentFindings.map(f => f.id));

    // Update existing findings
    for (const finding of currentFindings) {
      const existing = this.findings.get(finding.id);
      if (existing) {
        existing.lastSeenAt = cycle;
        if (finding.sourceHash !== existing.sourceHash) {
          existing.sourceHash = finding.sourceHash;
          existing.status = 'modified';
        } else if (existing.status !== 'closed') {
          // Same hash, still present — persistent
          existing.status = 'persistent';
        }
      } else {
        // New finding
        finding.firstSeenAt = cycle;
        finding.lastSeenAt = cycle;
        finding.status = 'pending';
        this.findings.set(finding.id, finding);
      }
    }

    // Findings that disappeared -> closed
    for (const [id, finding] of this.findings) {
      if (!currentIds.has(id) && finding.status !== 'closed') {
        finding.status = 'closed';
        finding.fixVerified = true;
      }
    }
  }

  /**
   * Get findings that have been persistent (same hash) for >= threshold cycles.
   */
  getPersistentFindings(minCycles: number = 3): FindingSignature[] {
    return Array.from(this.findings.values()).filter(f => {
      if (this.excludedIds.has(f.id)) return false;
      if (f.status !== 'persistent') return false;
      return (f.lastSeenAt - f.firstSeenAt) >= minCycles;
    });
  }

  /**
   * Get approaches that were tried but did not resolve the finding.
   */
  getFailedApproaches(): Array<{ findingId: string; approach: string }> {
    const result: Array<{ findingId: string; approach: string }> = [];
    for (const [findingId, attempts] of this.failedApproaches) {
      for (const attempt of attempts) {
        result.push({ findingId, approach: attempt.approach });
      }
    }
    return result;
  }

  /**
   * Mark a finding as excluded (architectural, cannot be fixed).
   * Excluded findings are removed from the score denominator.
   */
  excludeFinding(findingId: string): void {
    this.excludedIds.add(findingId);
    const finding = this.findings.get(findingId);
    if (finding) {
      finding.status = 'closed'; // Treated as resolved for scoring
      finding.fixVerified = true;
    }
  }

  /**
   * Record that a fix approach was attempted for a finding.
   */
  recordFixAttempt(findingId: string, cycle: number, approach: string): void {
    if (!this.failedApproaches.has(findingId)) {
      this.failedApproaches.set(findingId, []);
    }
    this.failedApproaches.get(findingId)!.push({ cycle, approach });
  }
}
```

### 10.3 verifyAndGate() — New Method on EvidenceStore

```typescript
/**
 * Verify the Merkle chain and apply the evidence gate.
 *
 * Walks all entries, verifies hash links and recomputation.
 * If passRate < 0.96, the gate fails.
 *
 * This method is called by GodLoopOrchestrator.verifyCycle().
 *
 * @returns  Gate result with pass/fail decision and detailed diagnostics
 */
async verifyAndGate(): Promise<{
  passed: boolean;
  passRate: number;
  brokenAt: number | null;
  details: EvidenceGateDetails;
}> {
  // Read all nodes from disk
  const nodes = this.readAllNodes();

  if (nodes.length === 0) {
    // Empty chain — trivially valid (first entry has not been written yet)
    return {
      passed: true,
      passRate: 1.0,
      brokenAt: null,
      details: {
        totalEntries: 0,
        validEntries: 0,
        invalidEntries: 0,
        firstBrokenEntry: null,
        chainIntact: true,
        hashMismatches: [],
      },
    };
  }

  let validCount = 0;
  const hashMismatches: Array<{ entryId: number; expected: string; actual: string }> = [];
  let firstBrokenEntry: number | null = null;

  for (let i = 0; i < nodes.length; i++) {
    const node = nodes[i];
    let isValid = true;

    // --- CHECK 1: Previous hash links correctly ---
    if (i === 0) {
      // Genesis node — prevHash must be all zeros
      if (node.prevHash !== '0'.repeat(64)) {
        isValid = false;
        hashMismatches.push({
          entryId: i,
          expected: '0'.repeat(64),
          actual: node.prevHash,
        });
      }
    } else {
      // Non-genesis — prevHash must match previous node hash
      const prevNode = nodes[i - 1];
      if (node.prevHash !== prevNode.hash) {
        isValid = false;
        hashMismatches.push({
          entryId: i,
          expected: prevNode.hash,
          actual: node.prevHash,
        });
      }
    }

    // --- CHECK 2: Current hash matches recomputation ---
    const recomputedHash = crypto
      .createHash('sha256')
      .update(node.prevHash + JSON.stringify(node.data), 'utf-8')
      .digest('hex');

    if (node.hash !== recomputedHash) {
      isValid = false;
      hashMismatches.push({
        entryId: i,
        expected: recomputedHash,
        actual: node.hash,
      });
    }

    if (isValid) {
      validCount++;
    } else if (firstBrokenEntry === null) {
      firstBrokenEntry = i;
    }
  }

  const totalEntries = nodes.length;
  const invalidEntries = totalEntries - validCount;
  const passRate = totalEntries > 0 ? validCount / totalEntries : 1.0;
  const passed = passRate >= 0.96;
  const chainIntact = invalidEntries === 0;

  // --- Tamper-proof audit trail ---
  // If gate failed, write evidence of the failure to the chain itself.
  // This creates an immutable record that the gate was checked and failed.
  if (!passed) {
    try {
      await this.append({
        phase: 'VERIFY' as GodLoopPhase,
        cycle: -1, // Indicates gate check, not a cycle
        score: -1,
        timestamp: new Date().toISOString(),
        action: `EVIDENCE GATE FAILED: passRate=${passRate.toFixed(4)}, ` +
          `brokenAt=${firstBrokenEntry}, invalid=${invalidEntries}`,
      });
    } catch (err: unknown) {
      // P3: Cannot write evidence of failure — log but continue
      const msg = err instanceof Error ? err.message : String(err);
      console.error(`[EVIDENCE_STORE] Failed to write gate-failure evidence: ${msg}`);
    }
  }

  return {
    passed,
    passRate,
    brokenAt: firstBrokenEntry,
    details: {
      totalEntries,
      validEntries: validCount,
      invalidEntries,
      firstBrokenEntry,
      chainIntact,
      hashMismatches,
    },
  };
}
```

```typescript
  t2Path: string | null;
```

```typescript
    `- Findings resolved: ${resolved}\n` +
    `- Findings modified: ${modified}\n` +
    `- Findings persistent: ${persistent}\n\n` +
    `## Critical Context for Wave ${state.wave + 1}\n` +
    `${content}\n`;
}
```

### 11.5 T2 Generation Triggers

T2 knowledge files are only generated when a milestone is reached. This prevents token budget waste on routine cycles.

| Trigger | T2 Content | Condition |
|---|---|---|
| Score improvement > 10 points | Architecture progress summary, what worked, what did not | `score - prevScore > 10` |
| Cycle completion (full loop) | Full cycle report, all findings resolved/persistent, approach analysis | Phase returns to AUDIT (cycle +1) |
| Wave completion (all agents returned) | Wave results, build quality assessment, recommendations | VERIFY phase passes |
| New critical finding introduced | Regression analysis, root cause, containment strategy | New critical finding in post-audit |
| Problem-solving output | Revised approach, key insights, decision chain | SOLVE phase completes |

```typescript
/**
 * Determine if a T2 milestone has been reached.
 * T2 is only generated when significant architectural
 * understanding changes — not on every cycle.
 */
private shouldGenerateT2(state: GodLoopState): boolean {
  const history = this.cycleTracker.getHistory();

  // Trigger 1: Score improvement > 10 points
  if (history.length >= 2) {
    const current = history[history.length - 1].score;
    const previous = history[history.length - 2].score;
    if (current - previous > 10) return true;
  }

  // Trigger 2: Cycle completion (full loop)
  if (state.phase === 'GAP' || state.phase === 'AUDIT') {
    return true;
  }

  // Trigger 3: New critical finding introduced
  const newCritical = state.currentFindings?.filter(
    f => f.severity === 'critical' && f.firstSeenAt === state.cycle,
  );
  if (newCritical && newCritical.length > 0) return true;

  // Trigger 4: Problem-solving output
  if (state.revisedApproach && state.revisedApproach.length > 0) {
    return true;
  }

  return false;
}
```

### 11.6 T1/T2 Ratio Specification

The ratio determines how much context budget goes to operational (T1) vs architectural (T2) context. It shifts based on cycle maturity:

| Cycle Range | T1 % | T2 % | Rationale |
|---|---|---|---|
| 1-3 (early) | 70% | 30% | More operational: agents need immediate instructions, not architecture |
| 4-10 (mid) | 60% | 40% | Balanced: agents need both operations and architecture |
| 11+ (late) | 50% | 50% | More architectural: deep problem-solving, root cause analysis |

```typescript
/**
 * Get the T1/T2 context budget ratio based on cycle number.
 *
 * Early cycles need more operational context (what to fix now).
 * Late cycles need more architectural context (why it is not fixing).
 */
private getContextRatio(cycle: number): { t1Percent: number; t2Percent: number } {
  if (cycle <= 3) {
    return { t1Percent: 0.70, t2Percent: 0.30 };
  }
  if (cycle <= 10) {
    return { t1Percent: 0.60, t2Percent: 0.40 };
  }
  return { t1Percent: 0.50, t2Percent: 0.50 };
}
```

### 11.7 Compaction Recovery Context Injection

When the conversation is compacted (context window fills up), the `compactingHook` fires. This hook reconstructs the Poseidon state from disk and injects a recovery message into the conversation — **not** the system prompt.

```typescript
/**
 * Compaction recovery for Poseidon Mode.
 *
 * On compaction event, this hook fires and:
 *   1. Reads current state.json from disk
 *   2. Reads latest T1 from .trident/god-loop/context/
 *   3. Reads latest T2 if a milestone was reached
 *   4. Constructs a recovery message
 *   5. Injects into messages (NOT system prompt)
 *
 * This ensures the God Loop survives compaction without losing state.
 */
function poseidonCompactionRecovery(targetPath: string): string | null {
  const statePath = path.join(targetPath, '.trident', 'god-loop', 'state.json');

  // P1: Defensive — verify state file exists
  if (!fs.existsSync(statePath)) {
    return null; // Not in Poseidon Mode
  }

  // P2: Type certainty — parse with validation
  let state: GodLoopState;
  try {
    state = JSON.parse(fs.readFileSync(statePath, 'utf-8'));
  } catch {
    return null; // Corrupt state — cannot recover
  }

  // Read T1 context
  let t1Content = '';
  if (state.t1ContextPath && fs.existsSync(state.t1ContextPath)) {
    t1Content = fs.readFileSync(state.t1ContextPath, 'utf-8');
  }

  // Read T2 context if available
  let t2Content = '';
  if (state.t2ContextPath && fs.existsSync(state.t2ContextPath)) {
    t2Content = fs.readFileSync(state.t2ContextPath, 'utf-8');
  }

  // Construct recovery message
  const recoveryMessage = `POSEIDON MODE ACTIVE. You are in Cycle ${state.cycle}, Phase ${state.phase}.
Score: ${state.score}/100. Wave: ${state.wave}.

${t1Content}

${t2Content ? '## Architectural Context\n' + t2Content + '\n' : ''}

Read .trident/god-loop/state.json for full state. Continue the GOD loop.
Do NOT stop until LOCKED or FAILED.

## NEXT ACTION
Call trident-poseidon action="start" targetPath="${targetPath}"`;

  return recoveryMessage;
}
```

The hook is registered in the session lifecycle:

```typescript
// In hook registration (eventHook):
hooks.on('compacting', (event: SessionEvent) => {
  const targetPath = event.metadata?.poseidonTargetPath;
  if (!targetPath) return; // Not in Poseidon Mode

  const recovery = poseidonCompactionRecovery(targetPath);
  if (recovery) {
    // Inject into messages, NOT system prompt
    event.injectMessage(recovery);
  }
});
```

### 12.3 Wave Manifest Generation (PLAN Phase)

The wave manifest is generated in five steps during the PLAN phase:

```typescript
/**
 * Generate a wave manifest from build spec and remaining findings.
 *
 * Steps:
 *   1. trident-deep-planning generates build spec from remaining findings
 *   2. Build spec parsed into individual agent tasks
 *   3. Tasks grouped by file (same file -> same agent)
 *   4. Dependencies resolved (if B imports A, B depends on A)
 *   5. Independent tasks assigned to same wave
 *
 * @param buildSpec          Output from trident-deep-planning
 * @param remainingFindings  Findings not yet resolved
 * @param cycle              Current cycle number
 * @param wave               Current wave number
 * @param targetPath         Project root
 * @returns                  WaveManifest for dispatch
 */
function generateWaveManifest(
  buildSpec: BuildSpec,
  remainingFindings: FindingSignature[],
  cycle: number,
  wave: number,
  targetPath: string,
): WaveManifest {
  // --- STEP 1: Already done (buildSpec passed in from deep-planning) ---

  // --- STEP 2: Parse build spec into individual tasks ---
  const tasks = parseBuildSpecToTasks(buildSpec, remainingFindings);

  // --- STEP 3: Group tasks by file ---
  const fileGroups = new Map<string, AgentTask[]>();
  for (const task of tasks) {
    const group = fileGroups.get(task.filePath) ?? [];
    group.push(task);
    fileGroups.set(task.filePath, group);
  }

  // --- STEP 4: Resolve dependencies ---
  const dependencies = resolveDependencies(fileGroups, targetPath);

  // --- STEP 5: Assign to wave (all independent tasks -> this wave) ---
  const agents: WaveAgent[] = [];
  let agentId = 1;

  for (const [filePath, taskGroup] of fileGroups) {
    // Skip agents whose dependencies are in this same wave
    // (they will be dispatched in a later wave)
    const agentDeps = dependencies.get(`agent-${String(agentId).padStart(3, '0')}`) ?? [];
    const hasSameWaveDep = agentDeps.some(depId => {
      // Check if depId maps to an agent already in this wave
      return fileGroups.has(getFilePathForAgent(depId, agents));
    });

    if (hasSameWaveDep) {
      continue; // Defer to next wave
    }

    const allFindings = taskGroup.flatMap(t => t.findings);
    const spec = taskGroup.map(t => t.description).join('\n');

    agents.push({
      id: `agent-${String(agentId).padStart(3, '0')}`,
      module: extractModuleName(filePath),
      filePath,
      spec,
      findings: allFindings,
      allowedTools: determineAllowedTools(allFindings),
      bashAccess: needsBashAccess(allFindings),
    });

    agentId++;
  }

  // --- Determine T1 context path ---
  const t1ContextPath = path.join(
    targetPath, '.trident', 'god-loop', 'context',
    `wave-${wave}-T1.md`,
  );

  return {
    cycle,
    wave,
    targetPath,
    agents,
    dependencies,
    t1ContextPath,
    createdAt: Date.now(),
  };
}
```

### 12.4 Dependency Resolution

Dependencies are resolved by analyzing import graphs. If agent B file imports from agent A file, B depends on A.

```typescript
/**
 * Resolve dependencies between agents based on import graph.
 *
 * If agent B file imports from agent A file, B depends on A.
 * This prevents agents from modifying files that depend on
 * not-yet-modified interfaces.
 *
 * @param fileGroups  Map of filePath -> tasks
 * @param targetPath  Project root
 * @returns           Dependency graph: agentId -> [dependency agentIds]
 */
function resolveDependencies(
  fileGroups: Map<string, AgentTask[]>,
  targetPath: string,
): Map<string, string[]> {
  const dependencies = new Map<string, string[]>();
  const filePaths = Array.from(fileGroups.keys());

  // Build a reverse index: filePath -> agentId
  const fileToAgent = new Map<string, string>();
  let id = 1;
  for (const filePath of filePaths) {
    fileToAgent.set(filePath, `agent-${String(id).padStart(3, '0')}`);
    id++;
  }

  // For each file, check its imports
  for (const filePath of filePaths) {
    const agentId = fileToAgent.get(filePath)!;
    const deps: string[] = [];

    const imports = extractImports(path.join(targetPath, filePath));
    for (const importPath of imports) {
      const resolvedImport = resolveImportPath(importPath, filePath, targetPath);
      const depAgentId = fileToAgent.get(resolvedImport);
      if (depAgentId && depAgentId !== agentId) {
        deps.push(depAgentId);
      }
    }

    dependencies.set(agentId, deps);
  }

  return dependencies;
}
```

### 12.5 Wave Manifest Usage in DISPATCH Phase

The wave manifest is consumed in the DISPATCH phase. The orchestrator reads it and generates explicit text instructions for the model. The model then dispatches agents by calling `task()` for each one.

```typescript
/**
 * Build dispatch instructions from a wave manifest.
 *
 * The MODEL reads these instructions and dispatches agents
 * by calling task() for each agent in ONE message.
 * The orchestrator does NOT dispatch agents itself.
 */
function buildDispatchInstructions(
  manifest: WaveManifest,
  targetPath: string,

  const agentBlocks = manifest.agents.map(agent => {
    const findingSummary = agent.findings.length > 0
      ? agent.findings.map(f => `  - ${f.id}: ${f.issue} (${f.severity}) at ${f.file}:${f.line}`).join('\n')
      : '  (none — architectural task)';

    return `### Agent ${agent.id}: ${agent.module}
- **File:** ${agent.filePath}
- **Task:** ${agent.spec}
- **Findings to address (${agent.findings.length}):**
${findingSummary}
- **Allowed tools:** ${agent.allowedTools.join(', ')}
- **Bash access:** ${agent.bashAccess ? 'YES' : 'NO'}
- **Context bridge:** Read T1 at ${manifest.t1ContextPath}`;
  }).join('\n\n');

  return `## WAVE ${manifest.wave} DISPATCH

Dispatch ${manifest.agents.length} agents in parallel by calling task() ${manifest.agents.length} times in ONE message.

${agentBlocks}

## AFTER ALL AGENTS RETURN
Call trident-poseidon action="start" targetPath="${targetPath}"
Phase will advance to VERIFY for evidence gate check.`;
}
```

### 12.6 Allowed Tools Determination

```typescript
/**
 * Determine which tools an agent is allowed to use based on
 * the findings it is addressing.
 *
 * @param findings  Findings assigned to this agent
 * @returns         List of allowed tool names
 */
function determineAllowedTools(findings: FindingSignature[]): string[] {
  const tools = ['read', 'grep', 'glob']; // Always allowed

  // Agents that fix source code need edit access
  const hasSourceFindings = findings.some(f => f.fileType === 'source');
  if (hasSourceFindings) {
    tools.push('edit', 'write');
  }

  // Agents with complexity findings may need to restructure
  const hasComplexityIssues = findings.some(
    f => f.issue.includes('COMPLEXITY') || f.issue.includes('LENGTH'),
  );
  if (hasComplexityIssues) {
    tools.push('write'); // Full rewrite access for restructuring
  }

  return tools;
}

/**
 * Determine if an agent needs bash access.
 * Bash access is granted for agents that need to compile or test.
 *
 * @param findings  Findings assigned to this agent
 * @returns         True if bash access is needed
 */
function needsBashAccess(findings: FindingSignature[]): boolean {
  // Agents fixing build-chain issues need bash to verify compilation
  return findings.some(f =>
    f.issue.includes('BUILD') ||
    f.issue.includes('COMPILE') ||
    f.issue.includes('IMPORT') ||
    f.issue.includes('DEPENDENCY'),
  );
}
```

### 12.8 Wave Manifest Validation

Before a wave manifest is used in DISPATCH, it is validated to ensure correctness:

```typescript
/**
 * Validate a wave manifest before dispatch.
 *
 * Checks:
 *   1. Every agent has at least one finding or a spec
 *   2. No two agents modify the same file
 *   3. All file paths are within targetPath (no escape)
 *   4. All dependency agent IDs exist in the manifest
 *
 * @param manifest    Wave manifest to validate
 * @returns           { valid: boolean, errors: string[] }
 */
function validateWaveManifest(manifest: WaveManifest): {
  valid: boolean;
  errors: string[];
} {
  const errors: string[] = [];

  // Check 1: Every agent has findings or spec
  for (const agent of manifest.agents) {
    if (agent.findings.length === 0 && (!agent.spec || agent.spec.trim().length === 0)) {
      errors.push(`Agent ${agent.id} has no findings and no spec`);
    }
  }

  // Check 2: No two agents modify the same file
  const filePaths = new Set<string>();
  for (const agent of manifest.agents) {
    if (filePaths.has(agent.filePath)) {
      errors.push(`Multiple agents modify ${agent.filePath}`);
    }
    filePaths.add(agent.filePath);
  }

  // Check 3: All file paths within targetPath
  for (const agent of manifest.agents) {
    const resolved = path.resolve(manifest.targetPath, agent.filePath);
    if (!resolved.startsWith(path.resolve(manifest.targetPath))) {
      errors.push(`Agent ${agent.id} file path escapes target: ${agent.filePath}`);
    }
  }

  // Check 4: All dependency IDs exist
  const agentIds = new Set(manifest.agents.map(a => a.id));
  for (const [agentId, deps] of manifest.dependencies) {
    if (!agentIds.has(agentId)) {
      errors.push(`Dependency references unknown agent: ${agentId}`);
    }
    for (const depId of deps) {
      if (!agentIds.has(depId)) {
        errors.push(`Agent ${agentId} depends on unknown agent: ${depId}`);
      }
    }
  }

  return { valid: errors.length === 0, errors };
}
```

## SECTION 13: THE POSEIDON ENFORCER HOOK

### 13.0 Purpose and Role

The `poseidon-enforcer-hook.ts` is the **model-independent enforcement layer** for Poseidon Mode. It is the **SAFETY NET**, not the driver.

**Critical distinction:**

| Component | Role | Analogy |
|---|---|---|
| `trident-poseidon` TOOL | **Driver** — orchestrates state machine, advances phases, decides next action | The train conductor |
| `poseidon-enforcer-hook.ts` HOOK | **Safety net** — detects derailment, injects advisory corrections | The guard rails |

The hook fires AFTER every tool execution. It reads state, checks whether the model called the right tool for the current phase, and injects advisory corrections if the model derailed. It NEVER blocks execution. It NEVER replaces the orchestrator. It NEVER throws.

### 13.1 Design Principles

```typescript
// ================================================================
// DESIGN PRINCIPLES — poseidon-enforcer-hook.ts
// ================================================================

// P1: NEVER replaces GodLoopOrchestrator
//     The trident-poseidon tool is ALWAYS the driver.
//     This hook is a passive observer + advisory injector.
//     It reads state. It does not write phase transitions.
//     The orchestrator owns state mutation.

// P2: NEVER injects dynamic content into system prompt
//     The system prompt injection is ONE STATIC LINE, byte-identical
//     every single turn. This preserves the LLM prompt cache prefix.
//     Dynamic content (phase, score, cycle) goes into MESSAGE injections,
//     which are appended to the conversation, not the system prompt.

// P3: NEVER blocks tool execution
//     The hook returns advisory objects: { messageInjection, warning }.
//     The opencode runtime decides whether to inject these.
//     The hook does NOT throw, does NOT reject, does NOT intercept.
//     Enforcement is advisory. State advancement is mechanical.

// P4: NEVER throws exceptions
//     Every operation is wrapped in try/catch.
//     File read fails -> return {} (no enforcement possible).
//     JSON parse fails -> return {}.
//     State write fails -> log, continue.
//     The hook is crash-proof by construction.

// P5: IS model-independent
//     The hook does not rely on the model being smart.
//     It checks a simple mapping: phase -> expected tools.
//     If the model called the wrong tool, it injects a correction.
//     Works equally with GPT-4, Claude, DeepSeek, or a random model.

// P6: IS ~120-150 lines total
//     NOT 430 lines. NOT a mini-orchestrator.
//     It reads state, compares tool, injects message. That is it.
//     Bloat is a defect. If it exceeds 150 lines, refactor.

// P7: IS registered as the 9th hook in trident-hooks.ts
//     After: eventHook, chatMessageHook, toolBeforeHook, toolAfterHook,
//     systemTransformHook, messagesTransformHook, compactingHook,
//     commandExecuteHook.
//     Position 9. Runs last in the hook chain.
```

### 13.2 Full TypeScript Implementation Specification

```typescript
// src/hooks/poseidon-enforcer-hook.ts
// ~120-150 lines. Model-independent enforcement for Poseidon GOD loop.
// This hook is the SAFETY NET, not the driver.
// The trident-poseidon TOOL is the driver. This hook catches derailments.

import * as fs from 'node:fs';
import * as path from 'node:path';

// [P7] No hardcoded paths — resolve dynamically
const STATE_DIR = path.join(process.cwd(), '.trident', 'god-loop');
const STATE_FILE = path.join(STATE_DIR, 'state.json');

// ---------------------------------------------------------------
// TYPE DEFINITIONS
// ---------------------------------------------------------------

interface GodLoopState {
  phase: string;
  cycle: number;
  score: number;
  targetScore: number;
  derailmentCount: number;
  successfulActions: number;
  active: boolean;
  sessionID: string;
  startedAt: number;
  lastPhaseChange: number;
  history: Array<{
    cycle: number;
    phase: string;
    score: number;
    timestamp: number;
    action: string;
  }>;
}

  systemInjection?: string;
  messageInjection?: string;
  warning?: string;
}

// ---------------------------------------------------------------
// PHASE-TO-EXPECTED-TOOL MAPPING
// ---------------------------------------------------------------
// This tells the hook what tool SHOULD be called in each phase.
// If the model calls a tool NOT in this list during the given phase,
// it is a potential derailment.

const PHASE_TOOL_MAP: Record<string, string[]> = {
  'INIT':           ['trident-poseidon'],
  'AUDIT':          ['trident-code-audit', 'trident-poseidon'],
  'SCORE':          ['trident-poseidon'],
  'DECIDE':         ['trident-poseidon'],
  'PLAN':           ['trident-deep-planning', 'trident-poseidon'],
  'DISPATCH':       ['task', 'trident-poseidon'],
  'COLLECT':        ['trident-context-synthesis', 'trident-poseidon'],
  'VERIFY':         ['trident-poseidon'],
  'AUDIT_RECHECK':  ['trident-code-audit', 'trident-poseidon'],
  'PROBLEM_SOLVE':  ['trident-problem-solving', 'trident-poseidon'],
  'LOCKED':         [],
  'FAILED':         [],
};

// ---------------------------------------------------------------
// STATIC SYSTEM PROMPT INJECTION
// ---------------------------------------------------------------
// ONE LINE. Byte-identical every turn. Preserves prompt cache prefix.
// This is appended at END of system array, never injected into the
// cached prefix. The model sees it every turn but the cache hash
// does not change because the prefix (everything before this line)
// remains identical.

const STATIC_POSEIDON_PROMPT =
  '[POSEIDON ACTIVE] Read .trident/god-loop/state.json for current phase. ' +
  'Execute the required action. Do NOT stop until LOCKED or FAILED.';

// ---------------------------------------------------------------
// POSEIDON ACTIVE CHECK
// ---------------------------------------------------------------

function isPoseidonActive(sessionID: string | undefined): boolean {
  // [P2] Validate input
  if (typeof sessionID !== 'string' || sessionID.length === 0) return false;

  // [P6] Verify state file exists
  if (!fs.existsSync(STATE_FILE)) return false;

  // [P1] Defensive read
  let state: GodLoopState;
  try {
    state = JSON.parse(fs.readFileSync(STATE_FILE, 'utf-8'));
  } catch {
    return false;
  }

  // Check if this session is the active Poseidon session
  return state.active === true && state.sessionID === sessionID;
}

// ---------------------------------------------------------------
// DERAILMENT COUNTER DECAY LOGIC
// ---------------------------------------------------------------
// Every 5 successful phase actions decrements derailment count by 1.
// Tracked in state.json: { derailmentCount: N, successfulActions: M }
// When successfulActions reaches 5:
//   derailmentCount = max(0, derailmentCount - 1)
//   successfulActions = 0
// Counter NEVER goes below 0.
// Counter NEVER resets to 0 from a non-zero value without 5 successes.

function applyCounterDecay(state: GodLoopState): GodLoopState {
  if (state.successfulActions >= 5) {
    state.derailmentCount = Math.max(0, state.derailmentCount - 1);
    state.successfulActions = 0;
  }
  return state;
}

// ---------------------------------------------------------------
// MAIN HOOK FUNCTION — fires on tool.execute.after
// ---------------------------------------------------------------

export function poseidonEnforcerAfter(
  input: { tool?: string; sessionID?: string; agent?: string },
  _output: unknown
): EnforcerResult {
  // [P2] Validate input
  if (!input?.tool || !input?.sessionID) return {};

  // [P6] Verify state file exists
  if (!fs.existsSync(STATE_FILE)) return {};

  // [P1] Defensive read
  let state: GodLoopState;
  try {
    state = JSON.parse(fs.readFileSync(STATE_FILE, 'utf-8'));
  } catch {
    return {}; // Can't read state, can't enforce
  }

  // Terminal states — no enforcement needed
  if (state.phase === 'LOCKED' || state.phase === 'FAILED') return {};

  // Check if Poseidon is active for this session
  if (!isPoseidonActive(input.sessionID)) return {};

  const expectedTools = PHASE_TOOL_MAP[state.phase] || [];
  const toolCalled = input.tool;

  // -------------------------------------------------------
  // CASE 1: Model called the CORRECT tool for this phase
  // -------------------------------------------------------
  if (expectedTools.includes(toolCalled)) {
    // CORRECT — increment successful actions for counter decay
    state.successfulActions = (state.successfulActions || 0) + 1;
    state = applyCounterDecay(state);

    // Persist updated state (fire-and-forget)
    try {
      fs.writeFileSync(STATE_FILE, JSON.stringify(state, null, 2));
    } catch {
      // Non-fatal — state write failed, continue
    }

    // No correction needed — model is on track
    return {};
  }

  // -------------------------------------------------------
  // CASE 2: Model called the WRONG tool — DERAILED
  // -------------------------------------------------------
  state.derailmentCount = (state.derailmentCount || 0) + 1;
  state.successfulActions = 0; // Reset success counter on derailment

  // Write updated derailment count (fire-and-forget)
  try {
    fs.writeFileSync(STATE_FILE, JSON.stringify(state, null, 2));
  } catch {
    // Non-fatal — state write failed, continue
  }

  // -------------------------------------------------------
  // ESCALATION LOGIC — graduated response
  // -------------------------------------------------------
  const dc = state.derailmentCount;
  const phase = state.phase;
  const score = state.score;
  const cycle = state.cycle;

  if (dc === 1) {
    // LEVEL 1: WARN — gentle correction
    return {
      messageInjection:
        '[POSEIDON ENFORCER] You are in POSEIDON MODE, phase ' + phase + '. ' +
        'Expected tool: ' + expectedTools.join(' or ') + '. ' +
        'You called: ' + toolCalled + '. ' +
        'Return to the GOD loop. ' +
        'Current score: ' + score + '/100, Cycle: ' + cycle + '.'
    };
  }

  if (dc === 2) {
    // LEVEL 2: BLOCK — stronger correction
    return {
      messageInjection:
        '[POSEIDON ENFORCER] SECOND DERAILMENT. ' +
        'Phase: ' + phase + '. ' +
        'REQUIRED ACTION: Call ' + expectedTools[0] + ' now. ' +
        'Score: ' + score + '/100. ' +
        'Do NOT perform any other action. The GOD loop must continue.',
      warning: 'Derailment #2 in session ' + input.sessionID
    };
  }

  if (dc === 3) {
    // LEVEL 3: RESTART — reset to last known good phase
    return {
      messageInjection:
        '[POSEIDON ENFORCER] THIRD DERAILMENT — RESTARTING PHASE. ' +
        'You have derailed 3 times from the GOD loop. ' +
        'Phase: ' + phase + '. Score: ' + score + '/100. Cycle: ' + cycle + '. ' +
        'MANDATORY: Call trident-poseidon action=status to read current state. ' +
        'Then execute the required action for phase ' + phase + '.'
    };
  }

  // dc >= 4: LEVEL 4 — LOCKOUT
  return {
    messageInjection:
      '[POSEIDON ENFORCER] LOCKOUT — ' + dc + ' derailments. ' +
      'The GOD loop is being managed mechanically. Read state.json. ' +
      'Phase: ' + phase + '. ' +
      'Execute ONLY the required action for this phase.'
  };
}
```

| Derailment Count | Level | Injection Type | Tone | Expected Model Response |
|---|---|---|---|---|
| 1 | WARN | `messageInjection` | Gentle reminder | Acknowledge, call correct tool |
| 2 | BLOCK | `messageInjection` + `warning` | Firm directive | Immediately call expected tool |
| 3 | RESTART | `messageInjection` | Force re-read state | Call `trident-poseidon action=status` |
| 4+ | LOCKOUT | `messageInjection` | Mechanical mandate | Execute ONLY required action |
| 5 successful actions | DECAY | (none) | (silent) | Counter decremented by 1 |

### 13.5 Counter Decay Logic

```typescript
// ================================================================
// DERAILMENT COUNTER DECAY
// ================================================================
//
// The derailment counter is not monotonically increasing.
// It decays when the model demonstrates consistent correct behavior.
//
// RULES:
// 1. Every correct tool call increments successfulActions.
// 2. When successfulActions reaches 5, decrement derailmentCount by 1.
// 3. Reset successfulActions to 0 after decay.
// 4. derailmentCount NEVER goes below 0.
// 5. A derailment resets successfulActions to 0 (corrective behavior
//    must be sustained, not intermittent).
//
// EXAMPLE SEQUENCE:
//   derailmentCount=0, successfulActions=0  (initial)
//   -> wrong tool -> derailmentCount=1, successfulActions=0
//   -> right tool -> derailmentCount=1, successfulActions=1
//   -> right tool -> derailmentCount=1, successfulActions=2
//   -> right tool -> derailmentCount=1, successfulActions=3
//   -> right tool -> derailmentCount=1, successfulActions=4
//   -> right tool -> derailmentCount=1, successfulActions=5 -> DECAY
//   -> derailmentCount=0, successfulActions=0
//   (model back to clean state)
//
// This prevents permanent escalation from a single bad turn.
// The model can earn back trust through sustained correct behavior.
```

## SECTION 14: AUDIT ENGINE IMPROVEMENTS

### 14.0 Problem Statement

The v4.4.0 audit engine suffers from **bundle pollution**: compiled bundle files (300KB-2MB of minified code) flood the audit with thousands of findings that:
1. Have no actionable fix (cannot fix a bundled file — fix the source)
2. Skew the score (1000+ bundle findings make 100 source findings look irrelevant)
3. Waste context budget (each finding consumes tokens)
4. Demoralize the GOD loop (score never improves because bundles never change)

v4.4.1 introduces six improvements to solve this.

### 14.1 Bundle Exclusion via First-Line Sentinel

Bundled files have telltale first lines. We read ONLY the first 256 bytes (one read, no full file load) and check for known bundle sentinels.

```typescript
// ================================================================
// BUNDLE DETECTION — First-Line Sentinel Check
// ================================================================
// Runs during the audit engine file collection step.
// Files identified as bundles are EXCLUDED from audit entirely.
// They do not appear in findings. They do not affect score.

interface BundleDetectionResult {
  isBundle: boolean;
  bundler?: 'bun' | 'esbuild' | 'rollup' | 'webpack' | 'minified-iife';
  sizeBytes: number;
}

function detectBundle(filePath: string): BundleDetectionResult {
  // [P2] Validate input
  if (typeof filePath !== 'string' || filePath.length === 0) {
    return { isBundle: false, sizeBytes: 0 };
  }

  // Check 1: File extension — only .js files are potential bundles
  if (!filePath.endsWith('.js')) {
    return { isBundle: false, sizeBytes: 0 };
  }

  // Check 2: File size — bundles are typically > 100KB
  let stat: fs.Stats;
  try {
    stat = fs.statSync(filePath);
  } catch {
    return { isBundle: false, sizeBytes: 0 }; // Cannot stat, do not exclude
  }

  if (stat.size < 100_000) {
    return { isBundle: false, sizeBytes: stat.size };
  }

  // Check 3: First-line sentinel (read only 256 bytes)
  try {
    const fd = fs.openSync(filePath, 'r');
    const buffer = Buffer.alloc(256);
    const bytesRead = fs.readSync(fd, buffer, 0, 256, 0);
    fs.closeSync(fd);

    const firstLine = buffer.toString('utf-8', 0, bytesRead).split('\n')[0];

    // Known bundle sentinels:
    const sentinels: Array<{
      pattern: string;
      bundler: BundleDetectionResult['bundler'];
    }> = [
      { pattern: '// @bun',       bundler: 'bun' },
      { pattern: '// @esbuild',   bundler: 'esbuild' },
      { pattern: '// rollup',     bundler: 'rollup' },
      { pattern: '// webpack',    bundler: 'webpack' },
      { pattern: '!function(',    bundler: 'minified-iife' },
      { pattern: '(function()',   bundler: 'minified-iife' },
      { pattern: '(function(',    bundler: 'minified-iife' },
    ];

    for (const sentinel of sentinels) {
      if (firstLine.startsWith(sentinel.pattern)) {
        return {
          isBundle: true,
          bundler: sentinel.bundler,
          sizeBytes: stat.size,
        };
      }
    }
  } catch {
    return { isBundle: false, sizeBytes: stat.size };
  }

  return { isBundle: false, sizeBytes: stat.size };
}

function isBundleFile(filePath: string): boolean {
  return detectBundle(filePath).isBundle;
}
```

### 14.2 .tridentignore Support

Projects can specify custom exclusion patterns via `.tridentignore`, using `.gitignore` syntax.

```typescript
// ================================================================
// .tridentignore — Custom Audit Exclusions
// ================================================================
// Same syntax as .gitignore.
// Read at start of audit scan.
// Matching paths excluded from scan entirely.
//
// DEFAULT .tridentignore:
// ----------------------------------------------
// # Bundled output
// dist/
// node_modules/
// build/
// out/
//
// # Generated artifacts
// .trident/
// .manta/
//
// # Source maps
// *.js.map
// *.min.js
//
// # Plugin directories (self-contained)
// plugins/manta/
// ----------------------------------------------

function loadTridentIgnore(targetPath: string): string[] {
  // [P2] Validate input
  if (typeof targetPath !== 'string' || targetPath.length === 0) return [];

  const ignoreFile = path.join(targetPath, '.tridentignore');
  if (!fs.existsSync(ignoreFile)) return [];

  try {
    const content = fs.readFileSync(ignoreFile, 'utf-8');
    return content
      .split('\n')
      .map((line) => line.trim())
      .filter((line) => line.length > 0 && !line.startsWith('#'));
  } catch {
    // [P3] Cannot read ignore file — do not crash
    return [];
  }
}

function isIgnored(filePath: string, patterns: string[]): boolean {
  // [P2] Validate inputs
  if (typeof filePath !== 'string' || !Array.isArray(patterns)) return false;

  for (const pattern of patterns) {
    if (typeof pattern !== 'string' || pattern.length === 0) continue;

    // Simple glob matching (same semantics as .gitignore basics)
    const regexStr = pattern
      .replace(/\./g, '\\.')
      .replace(/\*/g, '.*')
      .replace(/\//g, '(/|\\\\)');

    try {
      if (new RegExp(regexStr).test(filePath)) return true;
    } catch {
      continue;
    }
  }
  return false;
}
```

### 14.3 Confidence Floor

Findings below a confidence threshold are suppressed. They are still recorded in the audit artifact but are NOT dispatched to build agents and do NOT affect the score.

```typescript
// ================================================================
// CONFIDENCE FLOOR — Suppress Low-Confidence Findings
// ================================================================

const CONFIDENCE_FLOOR = 0.85; // 85 percent confidence required for dispatch

interface FindingSignature {
  id: string;
  file: string;
  line: number;
  severity: 'critical' | 'high' | 'medium' | 'low';
  confidence: number; // 0.0 to 1.0
  message: string;
  rule: string; // R0-R16 layer that produced it
  correction?: string;
  bundleFile?: boolean;
  suppressed?: boolean;
  suppressReason?: string;
}

interface ConfidenceFilterResult {
  dispatched: FindingSignature[];
  suppressed: FindingSignature[];
}

function filterByConfidence(
  findings: FindingSignature[]
): ConfidenceFilterResult {
  // [P2] Validate input
  if (!Array.isArray(findings)) {
    return { dispatched: [], suppressed: [] };
  }

  const dispatched: FindingSignature[] = [];
  const suppressed: FindingSignature[] = [];

  for (const finding of findings) {
    if (finding.confidence >= CONFIDENCE_FLOOR) {
      dispatched.push(finding);
    } else {
      finding.suppressed = true;
      finding.suppressReason =
        'confidence ' + finding.confidence.toFixed(2) +
        ' < floor ' + CONFIDENCE_FLOOR;
      suppressed.push(finding);
    }
  }

  return { dispatched, suppressed };
}
```

### 14.4 Call-Graph-Aware Deduplication

**Problem:** The audit reports "Enforcement function `getCurrentGate` is never called" — but `getCurrentGate` IS called, from a different file via import.

**Solution:** Build a cross-file call graph before reporting invocation integrity (R10) findings.

```typescript
// ================================================================
// CALL-GRAPH-AWARE DEDUPLICATION
// ================================================================
// Before R10 reports "function never called",
// it checks the cross-file call graph.
//
// A function is never called ONLY if:
// 1. It is not called within its own file (no direct invocation)
// 2. AND it is not exported (no export keyword)
// 3. OR it is exported but no other file imports it

interface CallGraph {
  imports: Map<string, Set<{ symbol: string; fromFile: string }>>;
  exports: Map<string, Set<string>>;
  reverseDeps: Map<string, Set<string>>;
}

function buildCallGraph(targetPath: string): CallGraph {
  const callGraph: CallGraph = {
    imports: new Map(),
    exports: new Map(),
    reverseDeps: new Map(),
  };

  const ignorePatterns = loadTridentIgnore(targetPath);
  const sourceFiles = collectSourceFiles(targetPath, '.ts', ignorePatterns);

  for (const filePath of sourceFiles) {
    let content: string;
    try {
      content = fs.readFileSync(filePath, 'utf-8');
    } catch {
      continue;
    }

    // Extract imports: import { X, Y } from './path'
    const importRegex =
      /import\s+(?:type\s+)?\{([^}]+)\}\s+from\s+['"]([^'"]+)['"]/g;
    let match: RegExpExecArray | null;
    while ((match = importRegex.exec(content)) !== null) {
      const symbols = match[1].split(',').map((s) => s.trim());
      const moduleSpecifier = match[2];
      const resolvedPath = resolveModulePath(
        filePath, moduleSpecifier, targetPath
      );

      const fileImports = callGraph.imports.get(filePath) || new Set();
      for (const symbol of symbols) {
        fileImports.add({
          symbol: symbol.split(' as ')[0].trim(),
          fromFile: resolvedPath,
        });
      }
      callGraph.imports.set(filePath, fileImports);
    }

    // Extract exports: export function X, export const X, export class X
    const exportRegex =
      /export\s+(?:async\s+)?(?:function|const|class|let|var)\s+(\w+)/g;
    while ((match = exportRegex.exec(content)) !== null) {
      const symbol = match[1];
      const fileExports = callGraph.exports.get(filePath) || new Set();
      fileExports.add(symbol);
      callGraph.exports.set(filePath, fileExports);
    }
  }

  // Build reverse dependency map
  for (const [, imports] of callGraph.imports) {
    for (const { symbol } of imports) {
      const dependents = callGraph.reverseDeps.get(symbol) || new Set();
      dependents.add('imported');
      callGraph.reverseDeps.set(symbol, dependents);
    }
  }

  return callGraph;
}

function isFunctionNeverCalled(
  functionName: string,
  sourceFile: string,
  callGraph: CallGraph
): boolean {
  const fileExports = callGraph.exports.get(sourceFile);
  const isExported = fileExports?.has(functionName) ?? false;

  if (isExported) {
    const dependents = callGraph.reverseDeps.get(functionName);
    if (dependents && dependents.size > 0) {
      return false; // Exported AND imported elsewhere
    }
  }

  return isExported;
}

function resolveModulePath(
  fromFile: string, specifier: string, _projectRoot: string
): string {
  if (specifier.startsWith('.')) {
    const dir = path.dirname(fromFile);
    const resolved = path.resolve(dir, specifier);
    for (const ext of ['.ts', '.js', '.tsx', '/index.ts', '/index.js']) {
      const candidate = resolved + ext;
      if (fs.existsSync(candidate)) return candidate;
    }
    return resolved;
  }
  return specifier;
}
```

```typescript
function validateFindingLocation(finding: FindingSignature): boolean {
```

### 14.7 Stalled Finding Detection

```typescript
// ================================================================
// STALLED FINDING DETECTION
// ================================================================
// A finding is stalled if it has persisted for 3+ consecutive cycles
// without being addressed.
// Stalled findings trigger PROBLEM_SOLVE phase in the GOD loop.

interface StalledFindingTracker {
  findingId: string;
  firstSeenCycle: number;
  lastSeenCycle: number;
  consecutiveCycles: number;
}

const stallThresholdCycles = 3;

function identifyStalledFindings(
  findings: FindingSignature[],
  tracker: Map<string, StalledFindingTracker>,
  currentCycle: number
): FindingSignature[] {
  const stalled: FindingSignature[] = [];

  for (const finding of findings) {
    if (finding.suppressed) continue;

    const tracked = tracker.get(finding.id);
    if (tracked) {
      if (currentCycle === tracked.lastSeenCycle + 1) {
        tracked.consecutiveCycles++;
        tracked.lastSeenCycle = currentCycle;
      } else {
        tracked.consecutiveCycles = 1;
        tracked.lastSeenCycle = currentCycle;
        tracked.firstSeenCycle = currentCycle;
      }
    } else {
      tracker.set(finding.id, {
        findingId: finding.id,
        firstSeenCycle: currentCycle,
        lastSeenCycle: currentCycle,
        consecutiveCycles: 1,
      });
    }

    const entry = tracker.get(finding.id);
    if (entry && entry.consecutiveCycles >= stallThresholdCycles) {
      stalled.push(finding);
    }
  }

  return stalled;
}
```

```typescript
6. DELIVERY: Write a summary of what you changed, what passed, what failed.
```

```typescript
// WAVE DISPATCH — Parallel Agent Spawning
// ================================================================

interface WaveManifest {
  cycle: number;
  waveNumber: number;
  findings: FindingSignature[];
  agentAssignments: Map<string, FindingSignature[]>;
}

function createWaveManifest(
  findings: FindingSignature[],
  cycle: number,
  waveNumber: number,
  maxAgentsPerWave: number = 3
): WaveManifest {
  const dispatched = findings.filter((f) => !f.suppressed && !f.bundleFile);
  if (dispatched.length === 0) {
    return { cycle, waveNumber, findings: [], agentAssignments: new Map() };
  }

  const assignments = new Map<string, FindingSignature[]>();
  const groups = groupFindingsByFile(dispatched);
  const agentCount = Math.min(maxAgentsPerWave, groups.length);
  const agentIds: string[] = [];
  for (let i = 0; i < agentCount; i++) {
    agentIds.push('trident_build_' + (i + 1));
  }

  let agentIdx = 0;
  for (const [, fileFindings] of groups) {
    const agentId = agentIds[agentIdx % agentIds.length];
    const existing = assignments.get(agentId) || [];
    assignments.set(agentId, [...existing, ...fileFindings]);
    agentIdx++;
  }

  return { cycle, waveNumber, findings: dispatched, agentAssignments: assignments };
}

function groupFindingsByFile(
  findings: FindingSignature[]
): Map<string, FindingSignature[]> {
  const groups = new Map<string, FindingSignature[]>();
  for (const finding of findings) {
    const existing = groups.get(finding.file) || [];
    existing.push(finding);
    groups.set(finding.file, existing);
  }
  return groups;
}
```

## SECTION 16: TIMEOUT RECALIBRATION

### 16.0 Problem Statement

v4.4.0 timeouts were calibrated for interactive single-turn usage (120 seconds for audit, 60 seconds for VLM). Poseidon Mode runs autonomously for 50+ cycles, each potentially involving a 30K LOC audit, multiple agent dispatches, and evidence collection. The v4.4.0 timeouts cause premature failures during legitimate long-running operations.

### 16.1 Current vs. Poseidon-Grade Timeout Table

| Timeout | File Location | v4.4.0 Value | v4.4.1 Value | Rationale |
|---|---|---|---|---|
| Audit timeout | `trident-tools.ts:259` | 120000ms (2min) | 3600000ms (60min) | 30K LOC audit takes 10-40 min |
| VLM timeout | `trident-vision.ts:136` | 60000ms (1min) | 600000ms (10min) | VLM analysis of large screenshots |

| Circuit breaker reset | `concurrency/index.ts` | 30000ms (30s) | 300000ms (5min) | Longer reset for stability |
| Token bucket capacity | `concurrency/index.ts` | 60 tokens | 600 tokens | Swarm dispatch burst capacity |
| Token bucket refill | `concurrency/index.ts` | 10/sec | 100/sec | Faster refill for parallel dispatch |

## SECTION 17: BUILD AGENT GATE CHAIN

### 17.0 Overview

Each `trident_build` agent operates its own internal 6-gate chain. This is separate from Poseidon's 10-phase GOD loop. The GOD loop dispatches agents; each agent runs its own gate chain internally.

```
POSEIDON GOD LOOP (10 phases)
    |
    +-- DISPATCH phase
    |       |
    |       +-- Agent 1: PLAN > BUILD > VERIFY > TEST > AUDIT > DELIVERY
    |       +-- Agent 2: PLAN > BUILD > VERIFY > TEST > AUDIT > DELIVERY
    |       +-- Agent 3: PLAN > BUILD > VERIFY > TEST > AUDIT > DELIVERY
    |
    +-- COLLECT phase (gather agent results)
    +-- ...
```

### 17.1 The 6 Gates

```
PLAN then BUILD then VERIFY then TEST then AUDIT then DELIVERY
```

| Gate | Purpose | Evidence Required | Pass Criteria | Fail Action |
|---|---|---|---|---|
| **PLAN** | Understand the work | Wave manifest read and parsed | Agent can articulate which findings to address and what fix each needs | Abort — re-read manifest |
| **BUILD** | Apply fixes | Files modified using edit/write tools | All specified findings have been addressed with actual code changes | Re-attempt failed findings |
| **VERIFY** | Confirm changes | Read-back of all modified files | Changed code matches intended fix | Revise and re-apply the edit |
| **TEST** | Compile and run | `tsc --noEmit` exit code 0 | Zero TypeScript compilation errors | Fix compilation errors |
| **AUDIT** | Self-check | Agent scans own changes for theatrical patterns | No empty catches, no hardcoded returns, no unchecked casts | Fix theatrical code, re-run TEST |
| **DELIVERY** | Report results | Summary report written to agent output | All findings reported with status (CLOSED, SKIPPED, FAILED) and evidence | Re-report with missing information |

```typescript
const GATE_ORDER: AgentGate[] = [
  'PLAN', 'BUILD', 'VERIFY', 'TEST', 'AUDIT', 'DELIVERY'

function advanceGate(chain: AgentGateChain): AgentGate | null {
  const currentIdx = GATE_ORDER.indexOf(chain.currentGate);
  if (currentIdx === -1) return null;
  if (currentIdx === GATE_ORDER.length - 1) return null;
  return GATE_ORDER[currentIdx + 1];
}

function canSkipGate(
  _chain: AgentGateChain,
  _targetGate: AgentGate

  // NO GATE CAN BE SKIPPED. This is a hard rule.
  return false;
}
```

```typescript
// ================================================================
// ORCHESTRATOR COLLECT PHASE — Gather Agent Reports
// ================================================================

function collectAgentReports(
  reports: AgentReport[],
  _previousScore: number,
  previousClosedCount: number
): {
  newScore: number;
  closedCount: number;
  closedThisCycle: number;
  failedAgents: string[];
  stalledFindings: string[];
} {
  let closedThisCycle = 0;
  const failedAgents: string[] = [];

  for (const report of reports) {
    if (!report.allGatesPassed) {
      failedAgents.push(report.agentId);
    }

    for (const finding of report.findingsAddressed) {
      if (finding.status === 'CLOSED') {
        closedThisCycle++;
      }
    }
  }

  const closedCount = previousClosedCount + closedThisCycle;

  return {
    newScore: 0, // Calculated by hash-based scoring engine
    closedCount,
    closedThisCycle,
    failedAgents,
    stalledFindings: [],
  };
}
```

### 17.7 Gate Chain vs. GOD Loop Comparison

| Aspect | Agent Gate Chain | GOD Loop |
|---|---|---|
| Scope | Single agent, single wave | Entire project, all cycles |
| Phases | 6 (PLAN to DELIVERY) | 10 (INIT to LOCKED) |
| Who runs it | trident_build agent internally | Orchestrator (main agent + tool) |
| State persistence | Agent report (ephemeral) | state.json (persistent) |

### FIX 1: Score Calculation — Real Hash Comparison

**Bug:** #1 — Score Stuck at 0

**Root Cause:** The `extractScore()` method relies on a regex match against the audit tool's text output. The audit tool always reports `Score: 0/100` because it has no semantic understanding of finding remediation. The regex therefore never finds a non-zero score, and the GOD loop believes zero progress has been made across all cycles.

**Current Behavior:**
- `extractScore()` reads `trident-code-audit` output text
- Regex `/Score:\s*(\d+)/` matches `Score: 0`
- Score stays at `0` forever
- `highestScore` never updates
- `stalledSince` increments every cycle
- Premature PROBLEM_SOLVE trigger at cycle 2
- LOCKED is never reached

**Fix Description:** Replace the regex-only `extractScore()` with `computeScore()` — a hash-based progressive scoring engine. The engine snapshots source context around each finding (file + line +/- 5 lines) before and after each wave, hashes the context, and compares hashes. A finding is counted as "closed" if its post-wave hash differs from its pre-wave hash. A finding is "modified" if partially changed. Score is the weighted ratio of closed+modified findings to total source findings.

**Exact File Path:** `src/poseidon/god-loop.ts`

**Before (Current extractScore — broken):**
```typescript

  return match ? parseInt(match[1], 10) : 0;
```

**After (Enhanced computeScore — fixed):**
```typescript
/**
 * computeScore — Hash-based progressive scoring engine.
 *
 * Replaces extractScore(). Instead of trusting audit tool text output
 * (which always reports 0), this method performs real content-hash
 * comparison between pre-wave and post-wave source context around each
 * finding.
 *
 * Scoring formula:
 *   score = ((closed + modified * 0.5) / totalSourceFindings) * 100
 *
 * Type weighting:
 *   .ts files: weight = 1.0
 *   .js files: weight = 0.1
 *
 * Confidence floor: only findings with confidence >= 0.85 are counted.
 *
 * @param postAuditFindings - Findings from the latest audit recheck
 * @param preWaveHashes     - Content hashes captured before the wave
 * @returns Score 0-100
 */
private computeScore(
  postAuditFindings: PoseidonFinding[],
  preWaveHashes: Map<string, string>,
): number {
  // P6: Dependency check — guard against empty inputs
  if (!postAuditFindings || postAuditFindings.length === 0) {
    return 0;
  }

  // Filter to source findings with confidence floor >= 0.85
  const sourceFindings = postAuditFindings.filter((f) => {
    // Exclude bundle files (sentinel check)
    if (this.isBundleFile(f.file)) return false;
    // Apply confidence floor
    if ((f.confidence ?? 0) < POSEIDON_CONFIG.scoring.confidenceFloor) {
      return false;
    }
    // Only source file types
    return f.file.endsWith('.ts') || f.file.endsWith('.js');
  });

  if (sourceFindings.length === 0) {
    return 0;
  }

  let closedCount = 0;
  let modifiedCount = 0;
  let totalWeight = 0;

  for (const finding of sourceFindings) {
    const weight = finding.file.endsWith('.ts') ? 1.0 : 0.1;
    totalWeight += weight;

    // Compute the finding's context key: file:line
    const contextKey = `${finding.file}:${finding.line}`;

    // Get the pre-wave hash for this finding's source context
    const preHash = preWaveHashes.get(contextKey);

    // Compute the post-wave hash by reading current source context
    const postHash = this.hashSourceContext(finding.file, finding.line);

    if (preHash === undefined || postHash === undefined) {
      // Cannot determine — skip this finding (P2: type certainty)
      continue;
    }

    if (postHash !== preHash) {
      // Hash changed — either closed or modified
      // Check if the finding still exists in post-audit results
      const stillExists = postAuditFindings.some(
        (f) => f.file === finding.file && f.line === finding.line && f.rule === finding.rule,
      );

      if (!stillExists) {
        closedCount += weight;
      } else {
        modifiedCount += weight * 0.5;
      }
    }
  }

  // P8: Config validation — guard against division by zero
  if (totalWeight === 0) {
    return 0;
  }

  const rawScore = ((closedCount + modifiedCount) / totalWeight) * 100;
  const clampedScore = Math.min(100, Math.max(0, Math.round(rawScore)));

  // Sanity check: compare against regex score if available
  // (used only for logging, does not override hash-based score)
  if (POSEIDON_CONFIG.scoring.logSanityCheck) {
    console.log(`[computeScore] Hash-based: ${clampedScore}, totalFindings: ${sourceFindings.length}`);
  }

  return clampedScore;
}

/**
 * hashSourceContext — Read +/- 5 lines around a finding and hash them.
 * Returns SHA-256 hash of the source context, or undefined on error.
 */
private hashSourceContext(filePath: string, line: number): string | undefined {
  try {
    // P1: Defensive import — verify module exists
    const fs = require('fs');
    const crypto = require('crypto');

    if (!fs.existsSync(filePath)) {
      return undefined;
    }

    const content = fs.readFileSync(filePath, 'utf-8');
    const lines = content.split('\n');
    const startLine = Math.max(0, line - 6); // +/- 5 lines (0-indexed)
    const endLine = Math.min(lines.length, line + 5);
    const context = lines.slice(startLine, endLine).join('\n');

    return crypto.createHash('sha256').update(context).digest('hex');
  } catch (err) {
    // P3: Error completeness — log and propagate undefined
    console.error(`[hashSourceContext] Error reading ${filePath}:${line}: ${err}`);
    return undefined;
  }
}

/**
 * isBundleFile — Check if a file path is a bundle artifact.
 * Bundle files are excluded from scoring (false positive elimination).
 */
private isBundleFile(filePath: string): boolean {
  const bundlePatterns = [
    /\/dist\//,
    /\/node_modules\//,
    /\.bundle\./,
    /\.min\./,
    /bundle\.js$/,
    /index\.js$/,
  ];
  return bundlePatterns.some((p) => p.test(filePath));
}
```

### FIX 2: Auto-Reaudit Trigger

**Bug:** #2 — No Re-Audit After Build

**Root Cause:** The `verifyCycle()` method checks build agent results (did they return OK?) but never re-runs the audit tool to detect whether findings were actually remediated. The model must manually decide to re-audit, which it frequently forgets to do, especially after long build waves.

```typescript
/**
 * verifyCycle — Verify build results AND auto-trigger re-audit.
 *
 * After dispatch wave completes:
 * 1. Validate all agents returned
 * 2. Run trident-code-audit against targetPath (mandatory)
 * 3. Capture fresh findings
 * 4. Call computeScore() with fresh findings + pre-wave hashes
 * 5. Update state.json score, highestScore, stalledSince
 * 6. Transition phase based on score
 *
 * @param results - Agent results from the dispatch wave
 */
private async verifyCycle(results: AgentResult[]): Promise<void> {
  // P5: Atomic state — snapshot before mutation
  const previousScore = this.state.score;
  const previousHighest = this.state.highestScore;

  // Step 1: Validate all agents returned
  const allReturned = results.length > 0 && results.every((r) => r !== undefined);
  if (!allReturned) {
    console.warn('[verifyCycle] Not all agents returned results');
    this.state.phase = 'PLAN';
    await this.saveState();
    return;
  }

  // Step 2: MANDATORY re-audit — capture fresh findings
  // P9: Async discipline — must await
  const auditResult = await this.runAudit(this.state.targetPath);

  if (!auditResult || !auditResult.findings) {
    // P3: Error completeness — log and propagate
    console.error('[verifyCycle] Re-audit returned no findings — cannot compute score');
    this.state.phase = 'PLAN';
    await this.saveState();
    return;

  // Step 3: Update post-audit findings in state
  this.state.postAuditFindings = auditResult.findings;

  // Step 4: Compute score with hash-based comparison
  const newScore = this.computeScore(
    auditResult.findings,
    this.state.preWaveHashes ?? new Map(),
  );

  // Step 5: Update state with new score
  this.state.score = newScore;

  if (newScore > previousHighest) {
    this.state.highestScore = newScore;
    this.state.stalledSince = 0; // Reset stall counter on improvement
  } else if (newScore <= previousHighest) {
    this.state.stalledSince++;
  }

  // Step 6: Transition phase based on score
  if (newScore >= POSEIDON_CONFIG.scoring.targetScore) {
    // Score target met — advance to evidence verification
    this.state.phase = 'VERIFY';
  } else if (this.state.stalledSince >= POSEIDON_CONFIG.stallDetection.maxStalledCycles) {
    // Stalled — trigger problem solving
    this.state.phase = 'PROBLEM_SOLVE';
  } else {
    // Not yet at target — re-plan for next cycle
    this.state.phase = 'AUDIT_RECHECK';
  }

  // Step 7: Log to audit trail
  await this.logAuditEvent('CYCLE_VERIFY', {
    cycle: this.state.cycle,
    previousScore,
    newScore,
    findingsCount: auditResult.findings.length,
    stalledSince: this.state.stalledSince,
  });

  // Step 8: Persist state
  await this.saveState();
}
```

**After (Fixed — four safeguards):**

```typescript
/**
 * generateFindings — Generate findings with false positive elimination.
 *
 * Safeguards:
 * 1. File length cross-reference (line must exist)
 * 2. Bundle file exclusion (sentinel check)
 * 3. .tridentignore support (custom exclusions)
 * 4. Confidence floor (>= 0.85 only)
 *
 * @param files - Scanned files to audit
 * @returns Filtered, validated findings
 */
function generateFindings(files: ScannedFile[]): Finding[] {
  const findings: Finding[] = [];

  // Load .tridentignore patterns
  const ignorePatterns = loadTridentIgnore();

  for (const file of files) {
    // Safeguard 2: Bundle file exclusion
    if (isBundleFile(file.path)) {
      continue;
    }

    // Safeguard 3: .tridentignore exclusion
    if (matchesIgnorePatterns(file.path, ignorePatterns)) {
      continue;
    }

    // P6: Dependency check — verify file has content
    if (!file.content || file.content.length === 0) {
      continue;
    }

    // Pre-compute line count for cross-reference
    const lineCount = file.content.split('\n').length;

    for (const rule of RULES) {
      // P2: Type certainty — validate rule
      if (!rule.pattern || !(rule.pattern instanceof RegExp)) {
        continue;
      }

      const matches = rule.pattern.exec(file.content);
      if (!matches) {
        continue;
      }

      const lineNumber = getLineNumber(file.content, matches.index);

      // Safeguard 1: File length cross-reference
      // P2: Type certainty — line must exist in file
      if (lineNumber > lineCount) {
        // Phantom finding — line doesn't exist
        continue;
      }

      const confidence = rule.confidence ?? 0.5;

      // Safeguard 4: Confidence floor
      if (confidence < POSEIDON_CONFIG.scoring.confidenceFloor) {
        continue;
      }
```

```typescript
 * 2. Exclude bundles, node_modules, dist, .trident

 * @returns Real cycle result with actual file and finding counts
 */
runFullCycle(targetPath: string): CycleResult {
  // P6: Dependency check — verify path exists
  if (!fs.existsSync(targetPath)) {
    // P3: Error completeness — return error, not fake numbers
    return {
      filesFound: 0,
      findings: 0,
      score: 0,
      error: `Target path does not exist: ${targetPath}`,
      timestamp: new Date().toISOString(),
    };
  }

  // Step 1: Scan for source files
  const sourceFiles = this.scanSourceFiles(targetPath);

  // Step 2: Run audit rules on each file
  const allFindings: Finding[] = [];
  for (const file of sourceFiles) {
    const fileFindings = this.auditFile(file);
    allFindings.push(...fileFindings);
  }

  // Step 3: Calculate score based on findings

  const warningCount = allFindings.filter((f) => f.severity === 'warning').length;

  // Score = 100 - weighted deduction
  const deduction = criticalCount * 10 + errorCount * 5 + warningCount * 1;
  const score = Math.max(0, Math.min(100, 100 - deduction));

  return {
    filesFound: sourceFiles.length,       // REAL count
    findings: allFindings.length,          // REAL count
    criticalFindings: criticalCount,
    errorFindings: errorCount,
    warningFindings: warningCount,
    score,
    timestamp: new Date().toISOString(),
  };
}

/**
 * scanSourceFiles — Recursively find .ts and .js files.
 * Excludes: node_modules, dist, .trident, bundles.
 */
private scanSourceFiles(targetPath: string): ScannedFile[] {
  const files: ScannedFile[] = [];
  const excludeDirs = ['node_modules', 'dist', '.trident', '.git'];
  const excludePatterns = [/\.bundle\./, /\.min\./, /bundle\.js$/];

  const scan = (dir: string) => {
    const entries = fs.readdirSync(dir, { withFileTypes: true });
    for (const entry of entries) {
      const fullPath = path.join(dir, entry.name);

      if (entry.isDirectory()) {
        if (!excludeDirs.includes(entry.name)) {
          scan(fullPath);
        }
      } else if (entry.isFile()) {
        // Only .ts and .js files
        if (entry.name.endsWith('.ts') || entry.name.endsWith('.js')) {
          // Exclude bundle patterns
          if (excludePatterns.some((p) => p.test(entry.name))) {
            continue;
          }
          try {
            const content = fs.readFileSync(fullPath, 'utf-8');
            files.push({ path: fullPath, content, lineCount: content.split('\n').length });
          } catch (err) {
            // P3: Error completeness — skip unreadable files
            console.warn(`[scanSourceFiles] Skipping ${fullPath}: ${err}`);
          }
        }
      }
    }
  };

  scan(targetPath);
  return files;
}

/**
 * auditFile — Run all audit rules on a single file.
 */
private auditFile(file: ScannedFile): Finding[] {
  const findings: Finding[] = [];
  const lines = file.content.split('\n');

  for (const rule of this.auditRules) {
    const regex = new RegExp(rule.pattern, 'g');
    let match;
    while ((match = regex.exec(file.content)) !== null) {
      const lineNumber = file.content.substring(0, match.index).split('\n').length;

      // P2: File length cross-reference
      if (lineNumber > lines.length) continue;

      // Confidence floor
      if ((rule.confidence ?? 0.5) < 0.85) continue;

      findings.push({
        file: file.path,
        line: lineNumber,
        rule: rule.name,
        severity: rule.severity,
        confidence: rule.confidence ?? 0.5,
      });
    }
  }

  return findings;
}
```

### 19.6 — T2 Milestone Detection Logic

```typescript
/**
 * shouldGenerateT2 — Determine if a T2 knowledge file should be generated.
 *
 * T2 is generated on significant milestones:
 * 1. Score improvement > 10 points
 * 2. Cycle completion (full loop)
 * 3. Wave completion with all agents returned
 * 4. New critical finding (regression detected)
 * 5. Problem-solving output
 */
function shouldGenerateT2(
  state: GodLoopState,
  previousState: GodLoopState,
): boolean {
  // 1. Score improvement > 10 points
  const scoreImprovement = state.score - (previousState.highestScore || 0);
  if (scoreImprovement >= POSEIDON_CONFIG.contextBridge.milestoneScoreImprovement) {
    return true;
  }

  // 2. Cycle completion (full AUDIT->...->DECIDE loop)
  if (state.cycle > previousState.cycle) {
    return true;
  }

  // 3. Wave completion with all agents returned
  if (state.phase === 'COLLECT' && previousState.phase === 'DISPATCH') {
    return true;
  }

  // 4. New critical finding introduced (regression)
  const newCriticals = state.postAuditFindings?.filter(
    (f) => f.severity === 'critical' && f.firstSeenAt === state.cycle,
  );
  if (newCriticals && newCriticals.length > 0) {
    return true;
  }

  // 5. Problem-solving output
  if (state.phase === 'PROBLEM_SOLVE') {
    return true;
  }

  return false;
}
```

## SECTION 20: THEATRICAL PURGING SYSTEM

> **Purpose:** Detect and purge theatrical code — functions that claim completion without performing real work. The system uses a 3-stage detection pipeline in the VERIFY phase, with escalating consequences for repeated violations.

### 20.1 — What Is Theatrical Code?

Theatrical code is code that **appears** to accomplish something but **actually does nothing**. Examples:

```typescript
// THEATRICAL: claims completion, performs no side effect
function applyFixes(): { outcome: 'ok' } {
  return { outcome: 'ok' }; // No file written, no change made
}

// THEATRICAL: claims runtime grade without verification
function verifyBuild(): { runtimeGrade: number } {
  return { runtimeGrade: 96 }; // No build run, no tests executed
}

// THEATRICAL: claims verified without checking evidence
function verifyEvidence(): { verified: boolean } {
  return { verified: true }; // No Merkle chain walked, no hashes checked
}
```

### 20.3 — Stage 2: Theatrical Return Detection (Side-Effect Tracing)

This is the core theatrical detection algorithm. For each function in modified files, it traces whether completion-claiming return statements are preceded by actual side-effect operations.

```typescript
/**
 * detectTheatricalReturns — Find functions that claim completion without side effects.
 *
 * Algorithm:
 * 1. Walk function body to find all return statements
 * 2. For each return with theatrical property (outcome ok, completed, verified, etc.)
 * 3. Walk BACKWARDS from the return statement
 * 4. Look for side-effect API calls:
 *    - fs.writeFileSync, fs.promises.writeFile
 *    - child_process.spawn, child_process.exec
 *    - fetch, http.request, https.request
 *    - database queries (INSERT, UPDATE, DELETE)
 *    - file renames, deletes, copies
 * 5. If NO side-effect call found before return -> THEATRICAL
 * 6. If side-effect call exists but appears AFTER return -> "side effect after claim"
 *
 * @param sourceFile - TypeScript source file AST
 * @returns Array of theatrical findings
 */
function detectTheatricalReturns(sourceFile: ts.SourceFile): TheatricalFinding[] {
  const findings: TheatricalFinding[] = [];

  // Theatrical return properties — keywords that claim completion
  const THEATRICAL_KEYWORDS = [
    'outcome',
    'completed',
    'verified',
    'done',
    'passed',
    'runtime grade',
    'runtimeGrade',
    'locked',
    'achieved',
  ];

  // Side-effect API patterns
  const SIDE_EFFECT_PATTERNS = [
    'writeFileSync',
    'writeFile',
    'fs.promises.writeFile',
    'spawn',
    'execSync',
    'exec',
    'fetch',
    'http.request',
    'https.request',
    'rename',
    'renameSync',
    'unlink',
    'unlinkSync',
    'copyFile',
    'copyFileSync',
    'mkdir',
    'mkdirSync',
    'INSERT',
    'UPDATE',
    'DELETE',
  ];

  // Visit all function declarations
  ts.forEachChild(sourceFile, function visit(node: ts.Node) {
    if (
      ts.isFunctionDeclaration(node) ||
      ts.isMethodDeclaration(node) ||
      ts.isArrowFunction(node)
    ) {
      // Analyze this function's returns
      const returns = findReturnStatements(node);

      for (const ret of returns) {
        // Check if return has theatrical property
        if (hasTheatricalProperty(ret, THEATRICAL_KEYWORDS)) {
          // Trace backwards for side effects
          const hasSideEffect = traceBackwardsForSideEffect(
            node,
            ret,
            SIDE_EFFECT_PATTERNS,
          );

          if (!hasSideEffect) {
            const lineNum =
              sourceFile.getLineAndCharacterOfPosition(ret.getStart()).line + 1;
            findings.push({
              file: sourceFile.fileName,
              line: lineNum,
              function: getFunctionName(node),
              type: 'theatrical_return',
              severity: 'critical',
              message: `Function "${getFunctionName(node)}" returns completion property ` +
                `without performing any side effect (no file write, no process spawn, ` +
                `no network call, no database operation found before return statement)`,
              principle: 'P10',
              suggestion: 'Add actual side-effect operation before returning completion, ' +
                'or change return to reflect actual work performed',
            });
          }
        }
      }
    }

    // Recurse into children
    ts.forEachChild(node, visit);
  });

  return findings;
}

/**
 * hasTheatricalProperty — Check if a return statement claims completion.
 */
function hasTheatricalProperty(
  returnStmt: ts.ReturnStatement,
  keywords: string[],
): boolean {
  if (!returnStmt.expression) return false;

  const text = returnStmt.expression.getText().toLowerCase();
  return keywords.some((kw) => text.includes(kw.toLowerCase()));
}

/**
 * traceBackwardsForSideEffect — Walk backwards from return to find side effects.
 */
function traceBackwardsForSideEffect(
  funcNode: ts.Node,
  returnStmt: ts.ReturnStatement,
  patterns: string[],
): boolean {
  // Collect all statements in the function before the return
  const statements: ts.Statement[] = [];

  function collectStatements(node: ts.Node) {
    if (ts.isBlock(node)) {
      for (const stmt of node.statements) {
        if (stmt.getStart() < returnStmt.getStart()) {
          statements.push(stmt);
        }
      }
      // Recurse into nested blocks
      ts.forEachChild(node, collectStatements);
    }
  }

  collectStatements(funcNode);

  // Check each statement for side-effect patterns
  for (const stmt of statements) {
    const text = stmt.getText();
    if (patterns.some((p) => text.includes(p))) {
      return true; // Side effect found
    }
  }

  return false; // No side effect found -> THEATRICAL
}
```

### 20.4 — Stage 3: Quarantine Mechanism

When CRITICAL theatrical findings are detected in the VERIFY phase, the offending file is quarantined — moved to a quarantine directory, excluded from compilation, and the wave is marked FAILED.

```typescript
const QUARANTINE_DIR = path.join(targetPath, '.trident', 'quarantine');

/**
 * quarantineFile — Move a theatrical file to quarantine.
 *
 * Actions:
 * 1. Move file to .trident/quarantine/
 * 2. Write metadata JSON with original path and reason
 * 3. Add quarantine to .gitignore if not present
 * 4. Mark wave as FAILED
 *
 * @param filePath - Path to the offending file
 * @param reason - Theatrical finding that triggered quarantine
 */
function quarantineFile(filePath: string, reason: TheatricalFinding): void {
  // P6: Dependency check — verify file exists
  if (!fs.existsSync(filePath)) {
    console.error(`[quarantineFile] File not found: ${filePath}`);
    return;
  }

  // P4: Resource lifecycle — ensure quarantine directory
  fs.mkdirSync(QUARANTINE_DIR, { recursive: true });

  const quarantinePath = path.join(QUARANTINE_DIR, path.basename(filePath));
  const metadataPath = quarantinePath + '.metadata.json';

  const metadata = {
    originalPath: filePath,
    quarantinePath: quarantinePath,
    reason: {
      type: reason.type,
      message: reason.message,
      function: reason.function,
      line: reason.line,
      principle: reason.principle,
    },
    quarantinedAt: new Date().toISOString(),
    cycle: currentCycle,
    wave: currentWave,
  };

  // Move file (atomic rename)
  fs.renameSync(filePath, quarantinePath);

  // Write metadata
  fs.writeFileSync(metadataPath, JSON.stringify(metadata, null, 2));

  // Ensure .gitignore excludes quarantine
  const gitignorePath = path.join(targetPath, '.gitignore');
  if (fs.existsSync(gitignorePath)) {
    const content = fs.readFileSync(gitignorePath, 'utf-8');
    if (!content.includes('.trident/quarantine/')) {
      fs.appendFileSync(gitignorePath, '\n.trident/quarantine/\n');
    }
  }

  console.error(`[QUARANTINE] File quarantined: ${filePath} -> ${quarantinePath}`);
  console.error(`[QUARANTINE] Reason: ${reason.message}`);
}
```

### 20.5 — Escalation Matrix

Theatrical violations escalate with repetition. Each agent's violation count is tracked across waves.

| Violation Count | Action | Description |
|----------------|--------|-------------|
| 1 | **WARN** | Log finding, allow wave to proceed. Add theatrical warning to T1 context for next wave so agents are aware of the pattern to avoid. |
| 2 | **BLOCK** | Block wave advancement. Re-dispatch with explicit anti-theatrical instructions: "Do NOT return completion without performing a side effect." |
| 3 | **QUARANTINE** | Move offending file to `.trident/quarantine/`. Mark wave as FAILED. Agent must re-dispatch with a new approach. |
| 4+ | **LOCKOUT** | Pause the agent. Trigger `trident-problem-solving` to analyze why theatrical patterns persist. Agent remains paused until problem-solving produces a revised approach. |

### 20.6 — Integration with VERIFY Phase

```typescript
// In the VERIFY phase handler:

case 'VERIFY': {
  // Step 1: Run evidence gate (Fix 3)
  const gateResult = evidenceStore.verifyAndGate();

  // Step 2: Run theatrical detection on all modified files
  const modifiedFiles = getModifiedFilesSince(lastWaveTimestamp);
  const theatricalFindings: TheatricalFinding[] = [];

  for (const file of modifiedFiles) {
    const sourceFile = parseTypeScript(file);
    const stage1 = runSemanticFirewall(sourceFile);     // Stage 1
    const stage2 = detectTheatricalReturns(sourceFile);  // Stage 2
    theatricalFindings.push(...stage1, ...stage2);
  }

  // Step 3: Escalate based on findings
  const criticalCount = theatricalFindings.filter((f) => f.severity === 'critical').length;

  if (criticalCount >= 3) {
    // QUARANTINE: Move files to quarantine
    for (const finding of theatricalFindings.filter((f) => f.severity === 'critical')) {
      quarantineFile(finding.file, finding);
    }
    this.state.phase = 'DISPATCH'; // Rewind
    this.state.waveFailed = true;
  } else if (criticalCount >= 1 && gateResult.passed) {
    // WARN or BLOCK depending on violation count
    handleTheatricalEscalation(theatricalFindings);
  }

  await this.saveState();
  break;
}
```
